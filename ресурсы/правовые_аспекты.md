## 1. Основные акты

| Юрисдикция | Акт | Что регулирует | Год |
|------------|-----|----------------|-----|
| 🇷🇺 РФ | **152-ФЗ «О персональных данных»** | сбор, хранение, передачу ПДн | 2006, ред. 2023 |
| 🇷🇺 РФ | **149-ФЗ «Об информации»** | общий правовой режим инфо | 2006 |
| 🇷🇺 РФ | **242-ФЗ** (поправка к 152-ФЗ) | локализация ПДн россиян в РФ | 2014 |
| 🇷🇺 РФ | **187-ФЗ «О безопасности КИИ»** | критическую инфраструктуру | 2017 |
| 🇪🇺 ЕС | **GDPR** | ПДн резидентов ЕС | 2018 |
| 🇪🇺 ЕС | **AI Act** | ИИ-системы по уровню риска | 2024 |
| 🇪🇺 ЕС | **Digital Services Act** | онлайн-платформы | 2022 |
| 🇺🇸 США | **CCPA (Калифорния)** | ПДн калифорнийцев | 2018 |
| 🇺🇸 США | **HIPAA** | медицинские данные | 1996 |

**Экстерриториальный принцип GDPR:** если ваша компания в РФ обрабатывает данные граждан ЕС (например, через сайт) — GDPR применим. Это часто игнорируют.

## 2. 152-ФЗ vs GDPR

| Параметр | 152-ФЗ | GDPR |
|----------|--------|------|
| Что считается ПДн | Любая информация, прямо или косвенно идентифицирующая физлицо | То же, но более детально |
| Согласие | Обязательно, может быть «общим» | Конкретное, информированное, отзываемое |
| Право на забвение | Ограниченно (через суд) | Прямо закреплено (Art. 17) |
| Право на портабельность | Нет явно | Есть (Art. 20) |
| Локализация | ДА: первичная запись ПДн россиян — в РФ | Нет (передача — по правилам трансграничности) |
| Уведомление об утечке | 24/72 часа в РКН | 72 часа в надзор |
| Штрафы (физлицу) | до 700 тыс. ₽ (с 2024) | до 4% мирового оборота / €20M |
| Уголовка | Возможна (ст. 137 УК РФ) | Через нац. право стран ЕС |

**Практический вывод:** даже если вы соблюдаете 152-ФЗ, для GDPR может не хватить — там более жёсткие требования к согласию и правам субъекта.

## 3. Категории данных

В Big Data важно с самого начала классифицировать данные. Это влияет на:
- Какие нормы применимы.
- Где данные можно хранить.
- Кому давать доступ.
- Сколько хранить.

### По 152-ФЗ

```
ПДн
├── Общие (ФИО, телефон, email, IP)
├── Биометрические (отпечатки, сетчатка, фото для распознавания)
├── Специальные (раса, политика, религия, здоровье, судимости)
└── Обезличенные (но потенциально восстановимые)
```

Биометрические и специальные категории требуют **отдельного письменного согласия** и почти всегда — обработки только в РФ.

### По GDPR (Article 9)

«Special categories» — раса, этническое происхождение, политические взгляды, религия, профсоюз, генетика, биометрия для идентификации, здоровье, сексуальная жизнь/ориентация.

Обработка этих данных по умолчанию **запрещена**, кроме узких исключений.

## 4. Обезличивание

**Это ключевое для Big Data.** Многие проблемы решаются обезличиванием **до** того, как данные попадают в кластер.

### 4.1. Псевдонимизация (pseudonymization)

Замена идентификатора (ФИО, email) на токен. При наличии «ключа» восстановление возможно.

**Пример в PySpark:**
```python
import hashlib
import pyspark.sql.functions as F

salt = "проект-2026-секретная-соль"

@F.udf("string")
def hash_email(email):
    return hashlib.sha256((salt + email).encode()).hexdigest()[:16]

df_pseudo = df.withColumn("user_token", hash_email(F.col("email"))).drop("email")
```

⚠️ **С точки зрения 152-ФЗ и GDPR псевдонимизированные данные — всё ещё ПДн!** Их защита обязательна.

### 4.2. Анонимизация (anonymization)

Необратимая. Восстановление невозможно ни при каких ключах. Только после полноценной анонимизации данные перестают быть ПДн.

Распространённые техники:
- **k-анонимность** — каждая запись неотличима от ещё k−1 записей.
- **l-разнообразие** — внутри групп есть разнообразие чувствительных значений.
- **Дифференциальная приватность** — добавление шума так, чтобы по выходу нельзя было восстановить вход.

⚠️ **Полная анонимизация трудно достижима.** Известные «деанонимизации» (Netflix Prize, AOL search logs) показали: одной только удалённой ФИО недостаточно — через комбинацию полей всё восстанавливается.

### 4.3. Правовое значение

```
ПДн → псевдонимизация → ПДн (просто защищённые)
ПДн → анонимизация    → НЕ ПДн (если действительно необратимо)
```

В Spark-пайплайне рекомендуется делать **обезличивание сразу после загрузки**, чтобы дальше с «чистыми» данными работали аналитики без особых прав.

## 5. Права субъекта

По GDPR субъект данных имеет права. Каждое право имеет техническое воплощение.

| Право | Техническая реализация |
|-------|------------------------|
| Доступ (Art. 15) | API/выгрузка всех данных пользователя |
| Исправление (Art. 16) | UPDATE по user_id во всех системах |
| Удаление / забвение (Art. 17) | DELETE из всех систем, ВКЛЮЧАЯ резервные копии и Parquet-партиции |
| Ограничение обработки (Art. 18) | флаг «не использовать в ML» |
| Портабельность (Art. 20) | выгрузка в машинно-читаемом формате (JSON) |
| Возражение против автомат. решений (Art. 22) | возможность запросить human review |

### 5.1. «Право на забвение» в Big Data — сложная штука

Вот почему:

1. Parquet-файлы **неизменяемы**. Нельзя «удалить строку», нужно перезаписать партицию.
2. Резервные копии живут долго.
3. ML-модель **«помнит»** обучающие данные опосредованно. Удаление из train-set не удаляет из модели → нужно переобучение.
4. В streaming-пайплайнах удалённое событие могло уже разойтись по downstream-системам.

Решения: **Delta Lake / Iceberg / Hudi** — таблицы с поддержкой DELETE и UPDATE поверх Parquet. → 01.4

### 5.2. Автоматизированные решения (Art. 22 GDPR)

Если ML-модель сама принимает решение, имеющее **юридические или существенные** последствия для человека (кредитный скоринг, отказ в страховке, фильтр в найме), субъект имеет право:

- знать о применении автоматики;
- получить пояснение;
- оспорить решение и потребовать review человеком.

Это — фундамент **объяснимого ИИ (XAI)**. → 06

## 6. AI Act ЕС (2024)

Систематизирует ML по уровню риска:

| Уровень | Что входит | Требования |
|---------|------------|------------|
| 🔴 Неприемлемый | Социальный рейтинг, манипуляция, real-time биометрия в публичных местах | **Запрет** |
| 🟠 Высокий | HR, кредитный скоринг, медицина, образование, юстиция | Conformity assessment, регистрация, документация |
| 🟡 Ограниченный | Чат-боты, deepfake | Прозрачность (предупреждение пользователю) |
| 🟢 Минимальный | Спам-фильтры, AI в играх | Без обязательств |

## 7. Доказательственная сила ML-выводов

Юристу полезно: насколько ML-вывод вообще можно использовать как доказательство?

В РФ — никакой специальной нормы нет, поэтому применяются общие правила:
- **АПК ст. 71 / ГПК ст. 67** — суд оценивает доказательства по внутреннему убеждению.
- **УПК ст. 88** — допустимость, относимость, достоверность.

Что обычно ослабляет ML-доказательство:
- Невозможность воспроизвести вывод (нет логов, кода, версии данных).
- Black-box характер модели (нет объяснения отдельного решения).
- «Грязные» обучающие данные (классовый/гендерный bias).
- Отсутствие сертификации/аттестации.

**Практика:** ML-вывод обычно используется как «вспомогательное» доказательство, а не самостоятельное. Сильнее, если есть:
- Версия модели + хэш весов.
- Версия данных + хэш.
- Лог входа/выхода для конкретного дела.
- Описание метрик качества (ROC-AUC, precision/recall).
- Метод объяснения (SHAP, LIME) для конкретного предсказания.

В модуле 06 мы будем учиться фиксировать всё это.

## 8. Чеклист для проекта

Перед запуском Big Data / ML проекта пройдитесь:

### Данные
- [ ] Что именно собирается? Это ПДн?
- [ ] Если ПДн — какая категория (общие, биометрические, специальные)?
- [ ] Есть ли согласие? Какое — общее или конкретное?
- [ ] Цель обработки **записана** и совпадает с реальностью?
- [ ] Срок хранения определён?

### Хранение
- [ ] Где физически лежат данные? РФ или нет?
- [ ] Резервные копии — где? Шифрование at rest?
- [ ] Доступы (ACL): кто может читать сырые ПДн?

### Обработка
- [ ] Обезличивание на каком шаге пайплайна?
- [ ] Audit log: кто и когда обращался к ПДн?

### ML
- [ ] Модель принимает автоматическое решение, влияющее на человека?
- [ ] Объяснимость: можете ли вы объяснить **отдельное** предсказание?
- [ ] Версионирование модели + данных?
- [ ] Метрики качества зафиксированы?
- [ ] Bias-тесты проводились?

### Безопасность
- [ ] Шифрование in transit (TLS)?
- [ ] Реакция на утечку: 24/72 ч в РКН + 72 ч надзору ЕС?

### Документы
- [ ] Политика обработки ПДн опубликована?
- [ ] Уведомление в РКН подано?
- [ ] DPIA (data protection impact assessment) сделан?
- [ ] Договор с подрядчиками (DPA)?
