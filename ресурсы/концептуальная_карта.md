## 1. Четыре «слоя» любой Big Data системы

Любая платформа (Hadoop on-prem, Spark в Databricks, BigQuery, AWS EMR) собирается из тех же блоков:

```
┌─────────────────────────────────────────────────────────────┐
│  4. ДОСТУП И ВИЗУАЛИЗАЦИЯ                                   │
│     BI-дашборды │ Notebook │ REST API модели │ Отчёты       │
├─────────────────────────────────────────────────────────────┤
│  3. ОБРАБОТКА И ML (compute)                                │
│     Spark Core │ Spark SQL │ Spark Streaming │ MLlib        │
│     MapReduce │ Hive │ Presto/Trino │ Flink                 │
├─────────────────────────────────────────────────────────────┤
│  2. ХРАНИЛИЩЕ (storage)                                     │
│     HDFS │ S3/MinIO │ HBase │ Cassandra │ Kafka topics      │
│     Форматы: CSV │ JSON │ Parquet │ ORC │ Avro              │
├─────────────────────────────────────────────────────────────┤
│  1. ИСТОЧНИКИ ДАННЫХ                                        │
│     OLTP БД │ Логи │ IoT │ API │ Файлы │ События            │
└─────────────────────────────────────────────────────────────┘
              ↕ сквозной слой ↕
   БЕЗОПАСНОСТЬ │ ПРАВО │ КАЧЕСТВО │ КАТАЛОГ │ МОНИТОРИНГ
```

Запомните: **хранение и обработка — это разные вещи**. Это ключевая идея Big Data в отличие от классических СУБД, где они слиты воедино.

## 2. Три парадигмы обработки

| Парадигма | Когда применять | Технологии | Латентность |
|-----------|------------------|------------|-------------|
| **Batch** | Раз в час/день, большие объёмы | Hadoop MR, Spark Batch | минуты-часы |
| **Streaming** | Каждое событие сразу | Spark Streaming, Flink, Kafka Streams | миллисекунды |
| **Interactive** | Аналитик пишет SQL | Hive, Presto, Spark SQL | секунды |

Многие современные системы — **гибрид** (lambda-архитектура, kappa-архитектура).

## 4. «Дерево решений» инженера данных

> Когда какую технологию выбирать. Учиться этому мы будем по ходу курса, но карту полезно увидеть сразу.

```
Размер данных?
├── < 1 ГБ ─────────────────────► Pandas. Spark тут ИЗБЫТОЧЕН.
├── 1 ГБ — 100 ГБ ──┐
│                   │ Поместится в RAM одной мощной машины?
│                   ├── Да ────► Pandas / DuckDB / Polars
│                   └── Нет ───► PySpark в local[*]
└── > 100 ГБ ─────────────────► PySpark / Hadoop, обязательно кластер

Какая скорость нужна?
├── Раз в день ────► Batch (Spark batch, Airflow)
├── Минуты ────────► Mini-batch (Spark Structured Streaming)
└── Секунды ───────► True streaming (Kafka + Flink/Spark Streaming)

Какой тип данных?
├── Чисто табличные ──► Spark SQL, Hive, Parquet
├── JSON / вложенные ──► Spark с schema-on-read, MongoDB
├── Графы ────────────► GraphX (Spark), Neo4j
└── Текст / NLP ──────► Spark + spaCy/HuggingFace
```

## 5. Жизненный цикл данных в Big Data

```
   1. Сбор    →  2. Приём (ingestion)  →  3. Хранение  →  4. Обработка
                                                              │
                                                              ▼
                  7. Использование  ←  6. Доступ  ←  5. Каталог
                  (BI, ML, отчёты)        (запросы)    (метаданные)
                              │
                              ▼
                      8. Жизненный конец
                      (архив / удаление)
```

Каждый этап имеет:
- **Технический аспект** (что делает софт).
- **Качественный аспект** (схема, профайлинг, тесты).
- **Правовой аспект** (согласие, законность хранения, срок).
- **Безопасностный аспект** (шифрование, ACL, аудит).

Юристу полезно держать в голове: **«где сейчас находятся данные?»** — потому что от ответа зависит, какие нормы применимы.

## 6. Мысленные модели, которые сильно помогают

### 6.1. «Везти руду к мельнице» vs «мельницу к руде»

В обычной СУБД мы тащим **данные к коду** (запрос вытаскивает строки, считает в памяти сервера). В Hadoop/Spark идея перевёрнута: мы тащим **код к данным** (вычисление едет к тому узлу, где лежит блок данных).

**Почему:** сетевой трафик дороже CPU. Если у вас 1 ПБ данных, перетянуть их по сети — нереально, а отправить 50 КБ кода — мгновенно.

### 6.2. Lazy = «не делай работу, пока не попросят результат»

Spark не выполняет ваши трансформации сразу. Он строит **план** и оптимизирует его. Реальный счёт начинается на **action** (`show`, `count`, `write`). Это позволяет:

- Удалять ненужные шаги (predicate pushdown).
- Объединять трансформации (Catalyst).
- Минимизировать перетасовку (shuffle).

### 6.3. Партиции — это «куски, по которым Spark распараллеливается»

Хороший партишенинг = быстрая обработка. Плохой = «один воркер пашет, остальные курят». В модуле 05 мы будем учиться видеть skew (перекос) и лечить его.

### 6.4. Shuffle — самая дорогая операция

Любой `join`, `groupBy`, `distinct` потенциально вызывает shuffle: данные передаются между узлами. Цель тюнинга — **минимизировать shuffle**.

## 7. Что НЕ является Big Data

Это полезно держать в голове, чтобы не «строить ракету, чтобы дойти до угла»:

- Один CSV на 200 МБ — это Pandas, а не Spark.
- Раз в неделю отчёт на 5 ГБ — это PostgreSQL + cron, а не Hadoop.
- 10 000 пользователей на сайте — это обычная база, а не Cassandra.

Современная мощная машина (64 ГБ RAM, NVMe) переваривает **до сотен гигабайт** без распределёнки. Это часто эффективнее кластера. Учитесь видеть, когда «большая» задача — на самом деле «средняя».
