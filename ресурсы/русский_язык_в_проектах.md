# 🇷🇺 Русский язык в Big Data проектах — гайд

> Какие части писать на русском, какие — на английском, и **почему**. Реальная практика российских команд.

---

## 1. Базовое правило

```
Код / API / технические термины → английский (мировой стандарт)
Документация / комментарии / UI / отчёты → русский (для пользователя)
Данные / категории / справочники → как удобно команде
```

Это **не** «всё переводить на русский». Это **разумное разделение**.

---

## 2. Что ВСЕГДА на английском

### 2.1. Spark / Pandas / Python API
```python
# ✅ Это норма
df.groupBy("category").agg(F.sum("amount"))

# ❌ Так не пишут
df.сгруппировать("категория").вычислить(F.сумма("сумма"))
```

Почему: вы пишете на Python. Все библиотеки — английские. Переименование разрушит совместимость с любой документацией, Stack Overflow, ChatGPT, ChatGPT-помощниками.

### 2.2. Имена функций и переменных
```python
# ✅ Профессионально
def clean_transactions(df):
    return df.dropDuplicates(["client_id"])

# ❌ Не стоит
def очистить_транзакции(df):
    return df.dropDuplicates(["id_клиента"])
```

В реальной команде через год придёт новый человек, не россиянин (например, удалённый). С английскими именами он войдёт в проект за день. С кириллическими — за месяц.

### 2.3. Названия колонок в технических таблицах
```sql
-- ✅ Стандарт индустрии
CREATE TABLE transactions (
    id BIGINT,
    client_id BIGINT,
    amount DECIMAL(15,2),
    ts TIMESTAMP,
    category VARCHAR(50)
);

-- ❌ Не делайте
CREATE TABLE транзакции (
    идентификатор BIGINT,
    идентификатор_клиента BIGINT,
    ...
);
```

Кириллические идентификаторы:
- Не работают в части SQL-движков.
- Требуют экранирования.
- Создают проблемы при экспорте в системы без UTF-8.
- Замедляют разработку.

### 2.4. Технические термины
```
DataFrame    — не «таблица данных».
Pipeline     — не «трубопровод».
Cluster      — не «гроздь».
Shuffle      — не «тасование».
Cache        — не «кэш-память».
ETL, MLOps, KPI, ROC-AUC, F1, batch, streaming, ...
```

Это **профессиональный жаргон**, его не переводят. Так и говорят: «у нас skew по одному ключу, надо salt'нуть».

## 3. Что МОЖНО (и нужно) на русском

### 3.1. Комментарии в коде
```python
# ✅ Объяснение бизнес-логики на русском
def calculate_credit_score(client):
    # Базовая формула из методики ЦБ РФ от 02.04.2024
    # https://internal-wiki/credit-scoring-methodology
    base = client.income / 30  # дневной доход
    if client.has_overdue:
        base *= 0.5  # штраф за просрочку — 50% (требование Compliance)
    return base
```

Бизнес-контекст пишите на русском — он понятнее команде. Технические комментарии можно по-английски.

### 3.2. Сообщения для пользователя (print, log, error)
```python
# ✅ Сообщения пользователю
log.info(f"Обработано {n:,} транзакций")
log.warning("Найдены подозрительные операции")

if not Path(input_file).exists():
    raise FileNotFoundError(f"Не найден входной файл: {input_file}")
```

Сообщения читают **не разработчики**, а аналитики, операторы, бизнес. Им нужен русский.

### 3.3. Документация
- README.md, ARCHITECTURE.md, CHANGELOG.md → русский (если команда русская).
- Docstrings к функциям → русский или английский, главное — **единообразно**.
- Model Card, Data Card → русский (для DPO, юр.отдела).

### 3.4. Данные с реальными значениями
```python
# ✅ Реальные русские категории
df["статус_договора"] = "Активный"
df["город"] = "Санкт-Петербург"

# ❌ Принудительная транслитерация без причины
df["status"] = "active"   # потерял исходную семантику
```

Если в исходных данных русские значения — **оставляйте**. Не транслитерируйте без надобности.

## 4. Гибридные паттерны

### 4.1. Колонки технические + значения с русскими

```python
# Типичный паттерн в РФ-проектах
df = spark.createDataFrame([
    (1, "Санкт-Петербург", "Активный", 12500.50),
    (2, "Москва", "Закрыт", 8400.00),
], ["client_id", "city", "status", "amount"])
```

Колонки на английском (для SQL/API), значения — реальные русские из данных.

### 4.2. Маппинг для интернационализации

Если ожидается выход в международный проект — храните **коды**, а перевод — в справочнике:

```python
# Таблица контрактов
contracts = spark.createDataFrame([
    (1, "ACTIVE", 100.0),
    (2, "CLOSED", 200.0),
], ["id", "status_code", "amount"])

# Справочник
statuses = spark.createDataFrame([
    ("ACTIVE", "Активный", "Active"),
    ("CLOSED", "Закрыт",   "Closed"),
], ["code", "ru", "en"])

# Локализованный вид
contracts.join(statuses, contracts.status_code == statuses.code) \
    .select("id", "ru", "amount")
```

## 5. Особенности кодировки

В РФ-проектах часто встречаются 3 кодировки:

| Кодировка | Где |
|-----------|-----|
| **UTF-8** | Современный стандарт. Всё новое — здесь. |
| **CP1251** | Старые Windows-системы, Excel экспорт, 1С. |
| **KOI8-R** | Очень старые системы, почти не встречается. |

При чтении CSV:
```python
# Не угадывайте — проверяйте
import chardet
with open("file.csv", "rb") as f:
    sample = f.read(10_000)
detected = chardet.detect(sample)
print(detected)   # {'encoding': 'windows-1251', 'confidence': 0.99, ...}

# Потом
df = pd.read_csv("file.csv", encoding="windows-1251")
```

Spark:
```python
df = spark.read.option("encoding", "windows-1251").csv("file.csv")
```

**Best practice:** всегда сохраняйте новое в UTF-8. Конвертируйте при загрузке.

## 6. Имена файлов и папок

```
# ✅ Норма в РФ-проектах
project/
├── README.md            (русский)
├── src/                  (английский)
│   ├── pipeline.py
│   └── utils.py
├── data/                 (английский)
└── docs/
    ├── архитектура.md    (русский) — для команды
    └── architecture.md    (английский) — для внешнего читателя
```

Имена директорий внутри `src/` — английские. Документация — на любом языке.

⚠️ Кириллица в путях:
- Большинство ОС поддерживают, но иногда возникают проблемы со старыми инструментами.
- Spark может ругаться на пути с пробелами и спецсимволами.
- В Git кириллица отображается правильно (с 2010-х).

**В нашем курсе** мы намеренно используем русские названия модулей (`01_основы_BigData/`, `ресурсы/шпаргалки/`) — для русскоязычной аудитории. Но в **production** проектах обычно — английские.

## 10. Анти-паттерны

### ❌ Транслитерация
```python
# Не делайте
df.select("klient_id", "summa", "kategoria")
```

Транслит работает плохо: «фамилия» → «familija»? «familiya»? «familiia»? Стандарта нет. Лучше английский эквивалент: `last_name`.

### ❌ Смешивание языков в одном слое
```python
# Хаос
def обработать_data(df):
    return df.фильтровать("amount > 0")
```

Или всё на английском, или (редко) всё на русском. **Не смешивайте**.

### ❌ Кириллица в коде «потому что хочется»
```python
class Клиент:   # ⚠️ работает, но...
    def __init__(self, имя: str):
        self.имя = имя
```

Это работает в Python. Но:
- Любая внешняя библиотека не поймёт.
- Code review усложняется.
- Stack Overflow / ChatGPT помогут плохо.

**Исключение:** учебные примеры (как в нашем курсе) — допустимо для наглядности.

## 11. Реальные практики компаний

### Тинькофф / Яндекс / Sber
- Код — английский.
- Документация — русский (с английскими секциями для международных команд).
- Pull request descriptions — обычно русский.
- Slack-каналы — русский с английскими терминами.

### СберДевайсы (международные проекты)
- Двуязычная документация.
- Код — английский.
- Дизайн-документы — английский (для патентов).

### Стартапы
- Часто **полностью на английском** — расчёт на скорый выход в международный рынок.
- Иногда комментарии на русском.

## 13. Совет для юриста на ML-команде

Если вы юрист и работаете с дата-командой:
- **Не настаивайте** на переводе всего кода на русский. Это технически не работает.
- **Настаивайте** на качественной документации на русском (для регулятора).
- **Просите** русскоязычные категории и метки данных, если конечный пользователь — русскоязычный.
- **Контролируйте** язык сообщений системы (log, error, dialogs) — пользователь должен понимать.
