## 2. SQL — задачки Junior

### Q1: «Топ-N продаж на товар»
```sql
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY product_id ORDER BY amount DESC) AS rn
    FROM sales
)
SELECT * FROM ranked WHERE rn <= 3;
```

### Q2: «Найти дубликаты по email»
```sql
SELECT email, COUNT(*) AS n
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

### Q3: «Пользователи без заказов за 30 дней»
```sql
SELECT u.*
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
  AND o.created_at > CURRENT_DATE - INTERVAL 30 DAY
WHERE o.user_id IS NULL;
```

### Q4: «Sessionization: новая сессия после 30 мин разрыва»
```sql
WITH events_with_gap AS (
    SELECT *,
        UNIX_TIMESTAMP(ts) - UNIX_TIMESTAMP(LAG(ts) OVER (PARTITION BY user_id ORDER BY ts)) AS gap
    FROM events
)
SELECT *,
    SUM(CASE WHEN gap > 1800 OR gap IS NULL THEN 1 ELSE 0 END)
        OVER (PARTITION BY user_id ORDER BY ts) AS session_id
FROM events_with_gap;
```

### Q5: «Cumulative sum с reset»
«Накопительная сумма, обнуляется в начале каждого месяца»
```sql
SELECT *,
    SUM(amount) OVER (
        PARTITION BY DATE_TRUNC('month', ts), user_id
        ORDER BY ts
    ) AS month_to_date
FROM transactions;
```

## 3. SQL — задачки Middle/Senior

### Q6: «Cohort retention»
```sql
WITH cohorts AS (
    SELECT user_id, DATE_TRUNC('month', MIN(signup_at)) AS cohort_month
    FROM users
    GROUP BY user_id
),
activity AS (
    SELECT
        c.cohort_month,
        DATE_TRUNC('month', e.ts) AS active_month,
        COUNT(DISTINCT e.user_id) AS active_users
    FROM events e
    JOIN cohorts c USING (user_id)
    GROUP BY c.cohort_month, DATE_TRUNC('month', e.ts)
)
SELECT
    cohort_month,
    active_month,
    active_users * 100.0 / FIRST_VALUE(active_users)
        OVER (PARTITION BY cohort_month ORDER BY active_month) AS retention_pct
FROM activity
ORDER BY cohort_month, active_month;
```

### Q7: «Funnel conversion»
«Прошёл ли пользователь шаги signup → view → cart → purchase»
```sql
WITH user_steps AS (
    SELECT user_id,
        MAX(CASE WHEN event = 'signup' THEN ts END) AS signup_at,
        MAX(CASE WHEN event = 'view' THEN ts END) AS view_at,
        MAX(CASE WHEN event = 'cart' THEN ts END) AS cart_at,
        MAX(CASE WHEN event = 'purchase' THEN ts END) AS purchase_at
    FROM events
    GROUP BY user_id
)
SELECT
    COUNT(*) AS total,
    COUNT(view_at) AS viewed,
    COUNT(cart_at) AS to_cart,
    COUNT(purchase_at) AS purchased,
    COUNT(purchase_at) * 100.0 / COUNT(*) AS conversion_rate
FROM user_steps
WHERE signup_at IS NOT NULL;
```

### Q8: «Найти 5-й по высоте элемент»
Классика. Решений много.

```sql
-- Через DENSE_RANK
SELECT salary FROM (
    SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
    FROM employees
) WHERE rk = 5;

-- Через LIMIT/OFFSET (быстрее, если нет дублей)
SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 4;
```

Подвох: спросят про дубли. Если зарплат 100, 100, 90 — что есть «второй»?

### Q9: «Median без percentile_approx»
```sql
WITH numbered AS (
    SELECT amount, ROW_NUMBER() OVER (ORDER BY amount) AS rn, COUNT(*) OVER () AS total
    FROM transactions
)
SELECT AVG(amount)
FROM numbered
WHERE rn IN ((total + 1) / 2, (total + 2) / 2);
```

### Q10: «Топ-N с обработкой ties»
«Топ-3 фильма за неделю, но если на 3-м и 4-м одинаковый рейтинг — оба»
```sql
SELECT *
FROM (
    SELECT *, DENSE_RANK() OVER (ORDER BY rating DESC) AS rk
    FROM films
)
WHERE rk <= 3;
```

## 4. Python coding на интервью

### Q11: «Two sum»
```python
def two_sum(nums: list[int], target: int) -> list[int]:
    seen = {}
    for i, n in enumerate(nums):
        if (target - n) in seen:
            return [seen[target - n], i]
        seen[n] = i
    return []
```

### Q12: «Word frequency на огромном файле»
```python
from collections import Counter

def top_n_words(file_path: str, n: int = 10) -> list[tuple[str, int]]:
    counter = Counter()
    with open(file_path) as f:
        for line in f:
            counter.update(line.lower().split())
    return counter.most_common(n)
```

Подвох: «Файл 100 ГБ, в память не помещается». → Нужно chunked чтение / MapReduce / Spark.

### Q13: «Удалить дубликаты из stream, не зная общего размера»
- Точно: `set()` — растёт неограниченно.
- Приближённо: **Bloom filter**.
- Гибрид: Bloom + точный set для подтверждения.

### Q14: «LRU кэш»
```python
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.d = OrderedDict()

    def get(self, key):
        if key not in self.d:
            return -1
        self.d.move_to_end(key)
        return self.d[key]

    def put(self, key, value):
        if key in self.d:
            self.d.move_to_end(key)
        self.d[key] = value
        if len(self.d) > self.cap:
            self.d.popitem(last=False)
```

## 5. Spark / Big Data вопросы

### Q15: «Что такое shuffle и почему он дорогой?»
Когда `groupBy` или `join`, Spark должен переместить данные с одинаковым ключом на один executor. Это:
- Сериализация → диск.
- Передача по сети.
- Десериализация.
- Merge-sort.
Цель оптимизации — минимизировать shuffle (broadcast, bucketing, аккуратные предикаты).

### Q16: «Когда broadcast join, а когда sort-merge?»
- **Broadcast** — если одна таблица < 10 МБ (по умолчанию). Маленькая копируется на executor'ы, нет shuffle.
- **Sort-merge** — если обе большие. Обе шафлятся по ключу, сортируются, merge-join.

Catalyst выбирает автоматически. Можно подсказать `broadcast()` или хинтом.

### Q17: «Что такое skew и как его лечить?»
Когда у одного ключа намного больше данных, чем у других → один executor работает часами, остальные простаивают.

Решения:
1. **AQE skewJoin** (Spark 3.0+) — автоматически.
2. **Salt-trick**: добавить случайный суффикс к горячему ключу, размножить вторую таблицу.
3. **Predicate filtering** перед join, если возможно.

### Q18: «Разница RDD vs DataFrame vs Dataset»
- **RDD** — низкий уровень, без схемы, без оптимизации.
- **DataFrame** — высокий уровень, со схемой, через Catalyst.
- **Dataset** — DataFrame + типизация (только Scala/Java).

В Python используем DataFrame.

### Q19: «Lazy evaluation — зачем?»
Spark не выполняет трансформации сразу — копит план. На action оптимизатор смотрит **весь pipeline** и переписывает (predicate pushdown, column pruning, объединение фильтров). Это даёт огромный выигрыш.

### Q20: «Как восстанавливается Spark при падении executor'а?»
Каждый DataFrame знает свою **lineage** (родословную). При потере партиции Spark пересчитывает её с последнего checkpoint'а. Если был cache — берёт из кэша.

## 6. Архитектура / System Design

### Q21: «Спроектируйте URL shortener (как bit.ly)»

Подвопросы:
- Как генерировать короткий URL? Hash от long? Counter + base62?
- Где хранить mapping? KV store (Redis/DynamoDB)?
- Чтения сильно больше записей → кэш L1 + L2.
- Как масштабировать на 1B URL? Sharding по hash.
- Аналитика: counter кликов → Kafka → Spark Streaming → агрегаты.

### Q22: «Спроектируйте систему рекомендаций»

- **Offline**: батч-сбор истории → embedding'и → top-K похожих (ANN, MinHash).
- **Online**: real-time inference, feature store, A/B tests.
- **Метрики**: CTR, конверсия, retention.
- **Cold start**: что показать новому пользователю?
- **Объяснимость**: «вы видели X, потому что...».

### Q23: «Спроектируйте систему детекции фрода»

- **Real-time или batch?** Обычно гибрид: real-time scoring + batch ML переобучение.
- **Источники**: Kafka transactions → Spark Streaming → ML inference.
- **Latency**: <100 мс.
- **False positive cost**: блокируем легитимного клиента — теряем его. Threshold важен.
- **Human-in-the-loop**: подозрительные → аналитик подтверждает / отклоняет → метки для переобучения.

### Q24: «Спроектируйте data lake для аналитики 100 ТБ/день»

- **Ingestion**: Kafka (для events), Sqoop/Debezium (для CDC из БД).
- **Storage**: S3 + Iceberg/Delta.
- **Compute**: Spark на EMR/Databricks; Trino для интерактивных запросов.
- **Catalog**: Hive Metastore / AWS Glue / Unity Catalog.
- **Orchestration**: Airflow.
- **BI**: Superset / Tableau поверх Trino.
- **ML**: feature store + MLflow.
- **Security**: encryption, IAM, audit log.

### Q25: «Спроектируйте Kafka для 1М events/sec»

- **Партиционирование**: достаточно партиций (CPU consumers'ов × 2-3).
- **Replication factor 3** — стандарт.
- **Retention** — зависит от use case (24ч для real-time, 7 дней для retry, навсегда для event sourcing).
- **Consumer groups** — масштабируются добавлением consumers (до числа партиций).
- **Backpressure**: что если consumer не успевает? Дискаунт events, alert, autoscale.

## 7. ML-вопросы

### Q26: «Bias-variance trade-off»
- High bias = underfitting (модель слишком простая).
- High variance = overfitting (слишком сложная, плохо обобщает).
- Решение: правильная сложность + регуляризация + больше данных.

### Q27: «Когда random forest лучше logistic regression?»
- Нелинейные зависимости.
- Много шумных признаков.
- Не нужна интерпретация коэффициентов.
- Хочется быстрого старта без feature engineering.

LR лучше при:
- Линейности.
- Большом количестве признаков (включая sparse).
- Нужна интерпретация.
- Распределённости.

### Q28: «Какие метрики выбирать для несбалансированных классов?»
Не Accuracy. Берите:
- **ROC-AUC** — не зависит от баланса.
- **PR-AUC** — лучше при сильном дисбалансе.
- **F1** — компромисс precision/recall.
- **Cost-sensitive metric** — если разная цена FP и FN (например, фрод).

### Q29: «Что такое data leakage и как избежать?»
Когда test-данные «протекают» в обучение, и модель показывает завышенные метрики, которые не воспроизведутся в production.

Примеры:
- Используете в features информацию, которой не будет на инференсе.
- Фит `StandardScaler` на всём датасете (включая test).
- Time series: будущее в обучении прошлого.

Защита:
- Train/test split **до** feature engineering.
- Pipeline (fit на train, transform на test).
- Time-based split для временных рядов.

### Q30: «Как мониторить ML модель в production?»
- **Data drift**: распределение входов меняется (Kolmogorov-Smirnov test, PSI).
- **Concept drift**: связь X→Y меняется (трекать качество на свежей разметке).
- **Performance**: latency, throughput.
- **Business metrics**: фрод-rate, conversion, выручка.
