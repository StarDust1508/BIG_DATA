# Урок 3.3 — MapReduce: парадигма

> Сама идея MapReduce — фундамент Big Data. Spark — это «MR в памяти». Понять MR = понять Spark на 80%.

📚 Концептуально мы это уже трогали в [уроке 1.3](../01_основы_BigData/урок_3_распределенные_вычисления.md). Здесь — глубже.

---

## Часть 1. Идея в одном абзаце

> Разбиваем задачу на две функции — **map** (обрабатывает одну запись) и **reduce** (агрегирует записи с одинаковым ключом). Между ними фреймворк сам сортирует и распределяет данные. Так можно посчитать что угодно на кластере любого размера.

---

## Часть 2. Этапы детально

```
INPUT
  │
  ├── Split: входной файл разбивается на «куски» (≈ блок HDFS)
  │
  ▼
MAP (параллельно на N машинах)
  │   Каждая строка → (key, value)
  │   Пример word count: "this is data" → ("this",1), ("is",1), ("data",1)
  │
  ▼
PARTITION + LOCAL SORT
  │   На каждом маппере результаты разбиваются по
  │   "partition" (hash(key) % N_reducers) и сортируются.
  │
  ▼
SHUFFLE (пересылка по сети)
  │   Все пары с одним ключом летят к одному редьюсеру.
  │
  ▼
SORT + GROUP
  │   На редьюсере данные сортируются и группируются по ключу.
  │
  ▼
REDUCE (параллельно на M машинах)
  │   Каждая группа (key, [v1, v2, ...]) → итоговое (key, agg)
  │
  ▼
OUTPUT
   Записывается в HDFS (по файлу на каждого редьюсера).
```

## Часть 3. Классический пример — WordCount

Задача: посчитать каждое слово в большом корпусе.

### Map
```python
# Вход:  строка текста
# Выход: пары (слово, 1)

for line in input:
    for word in line.lower().split():
        emit(word, 1)
```

### Shuffle (фреймворк)
```
("hello", 1) ─┐
("world", 1) ─┼─→ группируем по ключу:
("hello", 1) ─┘   "hello" → [1, 1]
                   "world" → [1]
```

### Reduce
```python
def reduce(word, counts):
    emit(word, sum(counts))
```

### Результат
```
hello   2
world   1
data    1
...
```

Что сделал фреймворк за вас:
- Раскинул входной файл по нодам.
- Запустил мапперы локально к данным.
- Перемешал пары по ключу (shuffle).
- Запустил редьюсеры на нужных нодах.
- Записал результат в HDFS.

Вы написали **только две функции**.

## Часть 4. Combiner — оптимизация

Если перед shuffle мы можем «частично свести» данные локально на маппере — это резко уменьшает сетевой трафик.

```
Без combiner:
  Mapper 1: ("hello", 1), ("hello", 1), ("hello", 1), ("hello", 1)
         → летят по сети 4 пары

С combiner на mapper'е:
  Mapper 1: combiner → ("hello", 4)
         → летит 1 пара
```

Combiner = мини-reducer, локальный на маппере. Часто можно использовать тот же код, что и в reduce (если операция **ассоциативна и коммутативна**, как `sum`, `max`, `min`).

Это идея «частичной агрегации» — основа эффективности и в Spark, и в современных движках.

---

## Часть 5. Что НЕ подходит для MR

Не все алгоритмы укладываются в map → reduce:

❌ **Итеративные** (большинство ML): K-means, gradient descent. Между итерациями MR пишет на диск → 10–100 итераций = 10–100 раз дисковые круги.
   → Спарк решает это in-memory.

❌ **Графовые** (PageRank): много итераций по соседям.
   → GraphX, Pregel, Spark GraphX.

❌ **Real-time / streaming**.
   → Spark Streaming, Flink, Kafka Streams.

✅ **Хорошо подходит:**
- Большие батчевые трансформации (ETL).
- Агрегации (count, sum по группам).
- Сортировки больших объёмов.
- Один шаг обработки.

Это объясняет, почему MR сегодня — нишевая технология: его обогнал Spark.

## Часть 6. Типы агрегаций в MR

Хорошие паттерны, которые работают одинаково в MR и Spark.

### 6.1. Group-by-агрегация
```
map:    record → (group_key, value)
reduce: (key, [values]) → (key, sum/avg/max)
```

### 6.2. Top-N
```
map:    record → (group_key, value)
reduce: (key, [values]) → (key, top_N(values))
```

### 6.3. Inner join двух датасетов
```
map:    record_A → (join_key, ("A", a_value))
        record_B → (join_key, ("B", b_value))
reduce: (key, [tagged_values]) → cross-product подходящих
```

### 6.4. Distinct
```
map:    record → (record, null)
reduce: (key, [null]) → (key, null)
```

### 6.5. Сортировка
MR гарантирует, что данные внутри **одного редьюсера** отсортированы по ключу. Глобальная сортировка возможна с **TotalOrderPartitioner**.

## Часть 7. Юридический угол

Все обработки в MR — это обработка данных. Если данные = ПДн:

- Запись результатов в HDFS должна быть в encryption zone.
- Если данные **сырые** ПДн, обезличивание должно произойти **в маппере** на самом раннем этапе.
- Logs и tmp-файлы MR-задачи — тоже хранилище ПДн → нужны audit log и периодическая очистка.

В реальной практике это редко делают вручную — обычно используют Spark с pandas-UDF / Spark MLlib pipeline, где обезличивание встроено в pipeline.
