## Часть 1. Из чего состоит Hadoop

Hadoop — это **три** независимых компонента, упакованных в один проект:

```
┌────────────────────────────────────────────────────┐
│                  HADOOP                            │
├────────────────────────────────────────────────────┤
│  HDFS       — распределённая файловая система      │
│  YARN       — менеджер ресурсов и приложений       │
│  MapReduce  — фреймворк compute (один из многих)   │
└────────────────────────────────────────────────────┘
```

Важно: **HDFS, YARN и MR — это разные вещи**. Часто видишь «работаю на Hadoop», и это может значить «храню в HDFS, считаю Spark'ом на YARN, MR не использую вообще». И это нормально.

## Часть 2. HDFS — Hadoop Distributed File System

### Идея

Файл размером 1 ТБ невозможно положить на один диск. Решение:

1. Разрезать файл на **блоки** (по умолчанию 128 МБ).
2. Раскидать блоки по разным машинам кластера.
3. Каждый блок **продублировать** на 3 машины (replication factor=3).
4. Хранить мета-данные «какой блок на какой машине» в одном месте.

```
Файл "big.csv" (1 ТБ)
   │
   ├── Block 1 → Node 2, Node 5, Node 8
   ├── Block 2 → Node 1, Node 4, Node 7
   ├── ...
   └── Block 8000 → Node 3, Node 6, Node 9
```

### Компоненты HDFS

```
┌────────────────────┐
│   NameNode         │  ← мозг: хранит метаданные
│   (один в HA — 2)  │     (file → blocks → nodes)
└─────────┬──────────┘
          │ управляет
┌─────────▼──────────┐  ┌──────────┐  ┌──────────┐
│   DataNode 1        │  │ DataNode2│  │ DataNode3│  ...
│   [blocks...]       │  │[blocks..]│  │[blocks..]│
└────────────────────┘  └──────────┘  └──────────┘
```

| Узел | Роль | Что хранит |
|------|------|------------|
| **NameNode** | Координатор | Мета-данные: путь файла → список блоков → IP-адреса DataNode'ов |
| **DataNode** | Хранилище | Реальные блоки данных |

⚠️ **NameNode — SPOF (single point of failure).** Если он упал, кластер недоступен. Решение: HA-режим с двумя NameNode'ами через ZooKeeper.

### Чтение файла из HDFS

```
1. Клиент → NameNode: «где блоки файла X?»
2. NameNode → Клиент: «блок 1 на Node 2/5/8, блок 2 на Node 1/4/7, ...»
3. Клиент → ближайшая DataNode → читает блок напрямую
```

NameNode видит только мета. Сами данные через него **не идут**. Это критично для производительности.

### Запись в HDFS

```
1. Клиент → NameNode: «хочу записать файл X»
2. NameNode → Клиент: «вот pipeline DataNode'ов»
3. Клиент → DataNode 1 → DataNode 2 → DataNode 3 (репликация)
4. По всем подтверждениям → клиент успех
```

## Часть 3. Размер блока: почему 128 МБ?

В обычной файловой системе блоки — 4–16 КБ. В HDFS — **128 МБ или 256 МБ**. Почему?

- Метаданные ограничены: 100 млн файлов × 128 МБ = 12.8 ПБ. Если бы блок был 4 КБ — у NameNode не хватило бы RAM.
- Каждый блок открывается отдельным сетевым соединением → крупные блоки = меньше overhead.
- Hadoop ориентирован на **последовательное чтение больших файлов**, а не на random access.

**Антипаттерн:** «много мелких файлов» — миллионы файлов по 1 КБ убьют NameNode и сделают чтение неэффективным. Решение: объединять (`hadoop archive`), писать в Parquet с укрупнёнными партициями.

## Часть 4. YARN — менеджер ресурсов

До Hadoop 2.0 «считать» можно было только MapReduce'ом. Это стало узким местом. В 2012 году вынесли менеджер ресурсов в отдельный слой — **YARN**.

```
┌─────────────────────────┐
│   ResourceManager (RM)   │ ← один на кластер
└───────────┬─────────────┘
            │ распределяет ресурсы
   ┌────────┼─────────┐
   │        │         │
┌──▼──┐  ┌──▼──┐  ┌──▼──┐
│NM 1 │  │NM 2 │  │NM 3 │   ← на каждой машине
│[cnt]│  │[cnt]│  │[cnt]│   ← запущенные контейнеры
└─────┘  └─────┘  └─────┘
```

| Компонент | Роль |
|-----------|------|
| **ResourceManager** | Принимает заявки приложений, выдаёт ресурсы |
| **NodeManager** | На каждом узле, запускает «контейнеры» |
| **ApplicationMaster** | Запускается на каждое приложение, координирует его таски |

Теперь поверх YARN работают: MapReduce, **Spark**, Tez, Flink, ...

> Если вы запускаете Spark на YARN — это и есть «Spark on Hadoop».

## Часть 5. MapReduce (compute)

Это парадигма + фреймворк. Парадигму разобрали в [уроке 1.3](../01_основы_BigData/урок_3_распределенные_вычисления.md). Здесь — практическая сторона.

```
┌─────────────────────────────────────────────────┐
│  Input  →  Map  →  Shuffle/Sort  →  Reduce → Out│
└─────────────────────────────────────────────────┘
```

Жизненный цикл MR-задачи на YARN:

1. Клиент отправляет JAR-файл с маппером/редьюсером в RM.
2. RM запускает **ApplicationMaster** в каком-то контейнере.
3. AM запрашивает у RM контейнеры для мапперов.
4. На каждом контейнере NM запускает map-таск (обычно — там, где лежит блок).
5. Map пишет промежуточный результат на **локальный диск**.
6. Shuffle перетягивает данные по сети к редьюсерам.
7. Редьюсеры пишут финал в HDFS.

**Главная боль MR:** между map и reduce — **запись на диск**. Это в 10–100 раз медленнее, чем держать в памяти. Поэтому Spark вытеснил MR.

## Часть 6. Локальность данных

Это идея, которую Spark унаследовал.

Hadoop старается **запускать таск там, где лежат блоки**:

```
Блок X лежит на Node 7 → запускаем map-task на Node 7 → читаем локально, без сети.
```

YARN при выделении контейнеров учитывает это через приоритеты:

| Уровень | Что значит |
|---------|------------|
| `NODE_LOCAL` | таск на той же ноде, что и блок (best) |
| `RACK_LOCAL` | на той же стойке (~1 GB/s в датацентре) |
| `ANY` | где угодно — приходится тащить по сети |

Поэтому большое число партиций ≈ большое число файлов в HDFS ≈ возможность параллелить + локальность.

## Часть 7. Когда Hadoop ещё актуален

Объективно: Hadoop постепенно уходит. Облака предлагают S3 + Spark + Lakehouse. Тем не менее:

✅ **Актуален:**
- Большие корпорации с on-prem кластерами и петабайтами данных.
- Регуляторика, не пускающая в облако (банки, госсектор).
- Уже существующие пайплайны на Hive/HBase.

❌ **Не актуален:**
- Новый проект «с нуля» в облаке (берите S3 + Spark, или Snowflake/BigQuery).
- Малые-средние объёмы (< 10 ТБ).
- Не нужен real-time.

В курсе мы трогаем Hadoop поверхностно — концептуально + одна MR-задача. Этого хватит, чтобы:
- Понимать настройки Spark (partitions, locality).
- Не теряться, если попадёте на проект с Hadoop.

## Часть 8. Юридический угол

HDFS — это файловая система. Все правовые принципы из [компендиума](../ресурсы/правовые_аспекты.md) применимы:

- **Шифрование at rest:** HDFS поддерживает encryption zones (kerberos + KMS).
- **ACL:** `hdfs dfs -chmod`, `hdfs dfs -chown` — но это POSIX-уровень. Для тонкого контроля — Apache Ranger или Sentry.
- **Локализация по 242-ФЗ:** если кластер физически в РФ — ОК. В облачном S3 — внимательно смотрите регион.
- **Right to be forgotten:** в голом HDFS — перезапись файла. В Hive поверх HDFS — то же самое + перестроение партиции. В Lakehouse (Hive 3+ ACID, Delta, Iceberg) — встроенный DELETE.
