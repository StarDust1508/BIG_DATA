## Часть 1. Где смотреть, что тормозит

### Spark UI
http://localhost:4040 (когда сессия активна).

Главные вкладки:
- **Jobs** — список job'ов.
- **Stages** — каждый job делится на stages. Здесь видно, какой stage тормозит.
- **SQL / DataFrame** — план запроса, время и rows на каждой стадии.
- **Storage** — что закэшировано.
- **Executors** — память и task'и на каждом.

⚠️ Если вы кодите в notebook — UI доступен только пока активна сессия. Сохраняйте скриншоты для дальнейшего анализа.

### explain()
```python
df.explain(True)
```
Покажет физический план. Что искать:
- `BroadcastHashJoin` (хорошо), `SortMergeJoin` (нормально), `BroadcastNestedLoopJoin` (плохо).
- `Exchange` — где shuffle.
- `Scan parquet` → `PushedFilters`, `ReadSchema`.

## Часть 2. Топ-причины медленности

### 2.1. Слишком мало или слишком много партиций
- Симптом: один executor молчит, остальные пыхтят (или наоборот).
- Решение: `repartition(N)` где `N ≈ 2–4 × ядер`.

### 2.2. Shuffle, которого можно избежать
- `groupBy` + `join` сразу → два shuffle. Иногда можно объединить.
- `orderBy` без `limit` → глобальный shuffle. Часто хватает `sortWithinPartitions`.

### 2.3. Skew (перекос по ключу)
- См. ниже §3.

### 2.4. Python UDF
- В 5–10× медленнее `F.*`. Заменяйте.

### 2.5. inferSchema
- Двойное чтение файла. Указывайте схему явно.

### 2.6. Множественные action'ы без cache
- Каждый `count()`/`show()` перевычисляет с нуля.
- Решение: `cache()` если используется > 1 раза.

### 2.7. Маленькие файлы
- Сотни тысяч мелких Parquet → читаются дольше, чем десяток крупных.
- Решение: `repartition` перед write.

### 2.8. Чтение CSV вместо Parquet
- CSV в 5–20× медленнее.
- Используйте Parquet в pipeline'ах.

## Часть 3. Skew в join — главная боль

### Симптомы

```
В Spark UI: Stage с 200 tasks. Из них 199 закончились за 10 секунд.
Один длится 30 минут.
```

Это означает, что у одного ключа — несоразмерно много данных.

### Диагностика

```python
# Распределение размеров партиций
df.groupBy(F.spark_partition_id()).count().describe().show()

# Топ ключей
df.groupBy("key").count().orderBy(F.col("count").desc()).show(10)
```

### Лечение 1: AQE skew join (бесплатно)
```python
spark.conf.set("spark.sql.adaptive.enabled", True)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)
```
AQE сам разделит «толстую» партицию.

### Лечение 2: salt-trick (для сложных случаев)
См. урок 5.3 «Joins».

### Лечение 3: предобработка
Сначала отделить «горячие» ключи в отдельный DataFrame, обработать broadcast'ом, потом union с остальными.

## Часть 4. Конфигурация Spark — что крутить

| Параметр | По умолчанию | Когда менять |
|----------|-------------|--------------|
| `spark.driver.memory` | 1g | Поднимать, если driver падает по памяти |
| `spark.executor.memory` | 1g | На реальных задачах — 4–8 ГБ |
| `spark.executor.cores` | 1 | 4–5 на executor — хороший компромисс |
| `spark.sql.shuffle.partitions` | 200 | Тюнить под размер данных |
| `spark.sql.autoBroadcastJoinThreshold` | 10 МБ | Поднимать, если есть «средние» таблицы (50–100 МБ) |
| `spark.sql.adaptive.enabled` | True (3.0+) | НЕ выключать |
| `spark.sql.adaptive.coalescePartitions.enabled` | True | НЕ выключать |
| `spark.sql.adaptive.skewJoin.enabled` | True | НЕ выключать |
| `spark.sql.sources.partitionOverwriteMode` | static | Ставьте `dynamic` для инкрементальных |

## Часть 5. Память executor'а

Память executor'а делится на:

```
spark.executor.memory = 8g
├── Reserved memory  (300 МБ) — для системных нужд
├── User memory     (40%) — UDF, переменные
└── Spark memory    (60%) — разделена на:
    ├── Storage (cache)        (50%)
    └── Execution (shuffle)     (50%)
```

Если `cache` забивает storage — execution не имеет места → spill на диск → медленно.

Решение:
- Не кэшируйте всё подряд.
- Используйте `StorageLevel.MEMORY_AND_DISK` (default) — spill на диск ОК.
- Перед записью `unpersist` ненужное.

## Часть 6. Диагностика OOM

`OutOfMemoryError` — самый частый «крик» в Spark.

**На driver'е** (`OutOfMemoryError: Java heap space`):
- `collect()` или `toPandas()` на больших данных.
- `broadcast()` слишком большой таблицы.

Лечение: не делайте collect; увеличьте `spark.driver.memory`.

**На executor'е:**
- Очень большая партиция.
- Skew + groupBy.

Лечение: увеличить `spark.executor.memory`; уменьшить размер партиций.

## Часть 7. Где НЕ нужно тюнить

Не тратьте время на тюнинг, если:
- Pipeline работает 30 секунд и его запускают раз в неделю.
- Объём данных небольшой.
- Машина новая, а вы пишете антипаттерны (Python for-loop на DataFrame'е).

Сначала исправьте **алгоритмические** проблемы. Потом — конфигурацию.
