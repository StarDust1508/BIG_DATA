# Урок 5.5 — Партиционирование: на диске и в памяти

> «Партиции» — слово, которое в Spark значит **две разные вещи**. Путаются почти все. Разбираем оба смысла.

---

## Часть 1. Два значения слова «партиция»

### 1. In-memory partition (Spark partition)
Кусок данных, который обрабатывается **одной task на одном core executor'а**.
- Управляется через `repartition()`, `coalesce()`.
- Не виден на диске.
- Это **runtime-параллелизм**.

### 2. On-disk partition (Hive-style partition)
Папка на диске вида `/year=2025/month=05/`. Реальное физическое разбиение.
- Управляется через `partitionBy()` при записи.
- Используется для partition pruning при чтении.
- Это **layout данных**.

```
В памяти: 16 in-memory partitions работают параллельно
   ↓ при write.partitionBy("year") ↓
На диске: 5 папок year=2021,...,2025 (по уникальным значениям)
```

---

## Часть 2. In-memory partitions: как они появляются

При чтении:
```python
df = spark.read.parquet("file.parquet")
df.rdd.getNumPartitions()    # обычно = количеству Parquet-файлов
```

При операциях:
- `read.csv` — по умолчанию ~128 МБ на партицию.
- После `groupBy/join` — `spark.sql.shuffle.partitions` (по умолчанию 200).
- Можно ручно: `df.repartition(N)` или `df.coalesce(N)`.

---

## Часть 3. repartition vs coalesce

```python
df.repartition(8)              # SHUFFLE → ровно 8 партиций ровного размера
df.repartition("city")          # SHUFFLE по hash(city)
df.repartition(8, "city")       # SHUFFLE по hash(city) → 8 партиций

df.coalesce(4)                  # БЕЗ shuffle → уменьшение партиций, неровно
```

| | repartition | coalesce |
|---|---|---|
| Shuffle | Да | Нет |
| Можно увеличить | Да | Нет (только уменьшить) |
| Скорость | Дорого | Дёшево |
| Размеры партиций | Равные | Неравные |

**Правило:**
- Хотите **больше** партиций → `repartition`.
- Хотите **меньше** партиций перед записью → `coalesce`.

---

## Часть 4. Сколько партиций нужно

Слишком мало → не используются ядра кластера.
Слишком много → overhead на запуск тасков.

**Эмпирическое правило:**
```
Партиций ≈ 2–4 × число ядер кластера
```

Размер одной партиции:
- Идеал: 100–500 МБ.
- Меньше 32 МБ — избыточный overhead.
- Больше 1 ГБ — risk OOM при операциях.

```python
spark.conf.set("spark.sql.shuffle.partitions", "200")    # по умолчанию
# Для маленьких кластеров (учёба) — 8-16
# Для больших — 1000+
```

С Spark 3.0+ AQE сам подбирает оптимально (`coalescePartitions.enabled`).

## Часть 5. Partitioning на диске (Hive-style)

```python
df.write \
  .mode("overwrite") \
  .partitionBy("year", "month") \
  .parquet("/data/transactions")
```

Создаст:
```
/data/transactions/
   year=2025/month=01/part-00000.parquet
   year=2025/month=01/part-00001.parquet
   year=2025/month=02/...
   year=2026/month=05/...
```

При чтении Spark знает о структуре:
```python
df = spark.read.parquet("/data/transactions/")
df.filter("year = 2026 AND month = 5").show()
# → читает ТОЛЬКО /year=2026/month=5/ (partition pruning)
```

## Часть 6. По чему партиционировать

Хорошие кандидаты:
- **Дата** (`dt`, `year`, `month`). Почти всегда.
- **Регион/страна** — если данных много и фильтр часто по региону.
- **Тип события** — если их немного (3–10).

Плохие кандидаты:
- **Уникальные ключи** (`client_id`, `transaction_id`). Миллионы папок!
- **С большим кардиналом** в целом.
- **Очень неравномерные** (один год — 1 ТБ, другой — 1 МБ → small files).

### Правила толщины партиций

| Стратегия | Размер партиции на диске |
|-----------|--------------------------|
| Идеал | 100 МБ – 1 ГБ |
| Слишком мало | < 50 МБ (small files problem) |
| Слишком много | > 5 ГБ (трудно перезаписывать) |

## Часть 7. Bucketing — фиксированное число «корзин»

Альтернатива партиционированию для **высококардинальных** колонок:

```python
df.write \
  .bucketBy(100, "user_id") \
  .sortBy("ts") \
  .saveAsTable("events_bucketed")
```

- Создаст **ровно 100 файлов** по `hash(user_id) % 100`.
- Spark знает, что одинаковые `user_id` лежат в одном бакете → **join без shuffle**.

⚠️ Bucketing работает только с `saveAsTable()` (Hive table), не с просто `.parquet()`.

### Когда брать

- Часто делаете join по `user_id` → bucketing совпадающих таблиц.
- Хотите стабильное число файлов независимо от данных.

---

## Часть 8. Dynamic partition overwrite

При перезаписи только конкретной партиции:

```python
spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic")

df.write \
  .mode("overwrite") \
  .partitionBy("dt") \
  .parquet("/data/tx")
```

Теперь `overwrite` затронет **только те `dt`**, что есть в `df`, а не сотрёт всё. Это must-have для инкрементальных пайплайнов.

---

## Часть 9. Small files problem в записи

Если у вас 200 in-memory партиций и `partitionBy("year","month")` с 12 значениями — получится **200 × 12 = 2400 файлов** в итоге. Каждый маленький — Spark будет тормозить при чтении.

Решение: **coalesce/repartition перед write**.

```python
df.repartition("year", "month") \
  .write.partitionBy("year", "month") \
  .parquet("/data/tx")
```

Теперь по одному файлу на каждую партицию.

Или контроль числа файлов на партицию:
```python
spark.conf.set("spark.sql.files.maxRecordsPerFile", 1_000_000)
```

## Часть 10. Юридический бонус

Партиционирование по дате — удобно для:
- **«Хранить 365 дней, потом удалять»**: просто rm партиции старше N дней.
- **Архивирование**: переместить старые партиции на медленное хранилище.
- **Audit log по дате**: ясно, что и когда писали.

Это техническая реализация принципа **storage limitation** из GDPR (Art. 5(1)(e)).
