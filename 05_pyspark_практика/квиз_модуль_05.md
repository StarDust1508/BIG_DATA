## Часть А. ETL и качество (6)

- c) Запускается параллельно

**2.** Что чаще даёт идемпотентность при записи?
- a) `mode("append")` + `partitionBy("dt")`
- b) `mode("overwrite")` + `partitionBy("dt")` + `partitionOverwriteMode=dynamic`
- c) `mode("ignore")`
- d) Запись в один файл

**3.** Где имеет смысл ставить DQ-проверки?
- a) Только в начале pipeline
- b) Только в конце
- c) После каждого критичного шага: extract, transform, перед записью

**4.** В Spark `null` и `NaN` для float-колонок:
- a) Одно и то же
- b) Разные значения; нужно проверять оба
- c) `NaN` запрещён
- d) Автоматически конвертируются

**5.** «Оставить самую свежую запись по client_id» делается через:
- a) `dropDuplicates(["client_id"])`

- c) `groupBy + agg(max)` — только если одна колонка
- d) Невозможно в Spark

**6.** Хорошая практика для ETL-функций:
- a) Одна большая функция с print'ами
- b) Чистые функции `(DataFrame) -> DataFrame`, легко тестируемые
- c) Делать всё в SQL-строке
- d) Кэшировать всё подряд

## Часть Б. Joins и windows (8)

**7.** Чтобы найти «транзакции «осиротевших» клиентов» (нет в clients):
- a) `tx.join(clients, "id", "inner")`
- b) `tx.join(clients, "id", "left_anti")`
- c) `tx.join(clients, "id", "right")`
- d) `tx.exceptAll(clients)`

**8.** `BroadcastHashJoin` применяется, когда:
- a) Обе таблицы большие
- b) Одна таблица < `spark.sql.autoBroadcastJoinThreshold` (~10 МБ)
- c) Используется RDD
- d) Streaming

**9.** Salt-trick применяется для:
- a) Обхода ошибки synthase
- b) Лечения skew по «горячему» ключу
- c) Шифрования
- d) Кэширования

**10.** Что вернёт `groupBy("client_id").agg(F.sum("amount"))` относительно исходного?
- a) Столько же строк
- b) По одной строке на клиента
- c) В 2 раза больше
- d) Один скаляр

**11.** Что нужно для «накопительной суммы по времени для каждого клиента»?
- a) `groupBy("client_id")`
- b) Window с `partitionBy("client_id").orderBy("ts")` и `sum().over(w)`
- c) `cumsum()` как в Pandas
- d) Только Pandas UDF

**12.** Чем `rank` отличается от `dense_rank`?
- a) Ничем
- b) `rank` оставляет пропуски при ничьих (1,1,3), `dense_rank` — нет (1,1,2)
- c) `rank` только для int
- d) `dense_rank` старее

**13.** Топ-3 транзакции каждого клиента — это:
- a) `groupBy("client_id").agg(F.top(3, "amount"))`
- b) `Window + row_number + filter rn<=3`
- c) `df.head(3 * n_clients)`
- d) Не сделать в Spark

**14.** Что НЕ делает skewed одну партицию опасной?
- a) Один executor пыхтит часами
- b) Возможный OOM на этом executor'е
- c) Job не закончится, пока эта таска не завершится
- d) Spark распараллелит её сам без AQE

## Часть В. Партиции, тюнинг, кэш (7)

**15.** «Партиция в памяти» — это:

- a) Coalesce делает shuffle
- b) Repartition нельзя использовать для уменьшения
- c) Coalesce НЕ делает shuffle, и может только уменьшать

**17.** Партиционировать на диске по `client_id` (миллион уникальных) — это:
- a) Отличная идея, ускорит чтение
- b) Антипаттерн: миллион папок, NameNode/файловая система загнутся
- c) Допустимо, если нужно
- d) Только для Delta

**18.** AQE `coalescePartitions.enabled` нужен, чтобы:
- a) Перепланировать join после shuffle
- b) Объединять мелкие партиции после shuffle в крупные
- c) Шифровать партиции
- d) Балансировать репликацию

**19.** `df.cache()` сразу триггерит вычисление?

**20.** Хороший «уровень кэша» по умолчанию для больших данных:

- b) `MEMORY_AND_DISK` (spill на диск, безопасно)
- c) `DISK_ONLY` (медленно)
- d) Без кэша всегда

**21.** Что НЕ помогает ускорить медленный Spark?
- a) Заменить CSV на Parquet

## Часть Г. Псевдонимизация и Delta (4)

**22.** Для хеширования ПДн в Spark предпочтительнее:
- a) Python UDF с hashlib.sha256
- b) `F.sha2(col, 256)` — встроенная, быстрая
- c) `F.md5` — короче
- d) Java RPC

**23.** Соль для хеширования должна храниться:
- a) В коде, на видном месте
- b) В Git
- c) В env / Secret Manager, отдельно от данных
- d) Где удобно

**24.** Delta Lake поверх Parquet добавляет:
- a) Сжатие
- b) ACID-транзакции, DELETE/UPDATE/MERGE, time travel, history
- c) Streaming-only поведение
- d) Запрет на чтение

**25.** «Право на забвение» (GDPR Art. 17) в Delta реализуется через:
- a) Перезапись всей таблицы
- b) `DELETE FROM delta.\`...\` WHERE client_id = X`
- c) Удаление NameNode'а
- d) Только вручную

7. **b.** left_anti.
8. **b.** Маленькая таблица — broadcast.
9. **b.** Salt-trick — против skew.
10. **b.** По строке на группу.
11. **b.** Window + sum.
12. **b.** Rank даёт пропуски, dense_rank — нет.
13. **b.** Window + row_number + filter.
14. **d.** Spark **не** распараллелит сам без AQE.
15. **b.** Партиция в памяти = единица параллелизма.

20. **b.** MEMORY_AND_DISK безопасен.
