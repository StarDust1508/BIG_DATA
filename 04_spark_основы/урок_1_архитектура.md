## Часть 1. Откуда взялся Spark

Spark родился в UC Berkeley AMPLab в 2009 году. Идея в одном предложении:

> Делаем то же, что MapReduce, но **в памяти**, и с **умным оптимизатором**.

В 2014 стал проектом Apache, в 2015 появился DataFrame API, в 2016+ — оптимизатор Catalyst и Tungsten. К 2020 году Spark стал де-факто стандартом распределённой обработки.

## Часть 2. Что такое Spark «один взгляд»

```
┌────────────────────────────────────────────┐
│  Ваш Python-код (PySpark)                  │
└──────────────────┬─────────────────────────┘
                   │ через py4j
┌──────────────────▼─────────────────────────┐
│  Spark Driver (JVM)                         │
│   - SparkSession                            │
│   - DAG-планировщик                          │
│   - Catalyst оптимизатор                     │
└──────────────────┬─────────────────────────┘
                   │ распределяет работу
   ┌───────────────┼───────────────┐
   ▼               ▼               ▼
┌─────┐         ┌─────┐         ┌─────┐
│ Ex1 │         │ Ex2 │         │ Ex3 │   ← Executors на разных нодах
│ JVM │         │ JVM │         │ JVM │
│data │         │data │         │data │
└─────┘         └─────┘         └─────┘
```

**Главное:**
1. **Driver** — управляющий процесс. Один на приложение.
2. **Executors** — рабочие. Их N, каждый — JVM-процесс на отдельной (обычно) машине.
3. **Кластер-менеджер** распределяет ресурсы: Standalone, YARN, Kubernetes, Mesos.

В режиме `local[*]` driver и executors — это потоки одного JVM-процесса на вашем ноутбуке. Учебный сценарий.

## Часть 3. Партиции — единица параллелизма

```
DataFrame
├── Partition 1 ──► обрабатывает Executor 1, core 1
├── Partition 2 ──► Executor 1, core 2
├── Partition 3 ──► Executor 2, core 1
├── Partition 4 ──► Executor 2, core 2
└── ...
```

**Партиция** = кусок данных, который обрабатывается **одной задачей (task)** на **одном ядре (core)** одного **executor'а**.

Это первое, что надо понять про Spark:
- Слишком мало партиций → не используются все ядра → медленно.
- Слишком много → overhead на запуск тасков → медленно.
- Идеал: ≈ 2–4 × числа ядер кластера.

## Часть 4. Что лежит «под капотом» каждого executor'а

```
Executor (JVM, типично 4 ядра / 8 ГБ RAM)
├── Task slots (= число ядер)
├── Storage memory (кэш)
├── Execution memory (shuffle, агрегации)
└── User memory (UDF, переменные)
```

Когда вы запускаете job:
1. Driver строит DAG (граф трансформаций).
2. DAG режется на **stages** (между shuffle).
3. Stage режется на **tasks** (по числу партиций).
4. Tasks отправляются на executors.
5. Executors считают и возвращают результат (или пишут на диск).

## Часть 5. Stage и shuffle — самое важное для производительности

```
Job
├── Stage 1   ┐
│   ├── Task 1 │  Эти таски выполняются параллельно.
│   ├── Task 2 │
│   └── Task N ┘
│       │
│       ▼  shuffle  (между stage 1 и stage 2 — пересылка по сети)
│       
├── Stage 2
│   ├── Task 1
│   └── ...
```

**Любой `groupBy`, `join`, `distinct`, `repartition`, `orderBy`** вызывает shuffle:
1. Stage 1 заканчивается, пишет результат на локальный диск каждого executor'а.
2. Executors стянут эти куски с других нод (network IO).
3. Stage 2 запускается.

**Shuffle — самая дорогая операция в Spark.** Цель тюнинга — минимизировать или избежать.

## Часть 6. Lazy evaluation

Это, возможно, **самое необычное** в Spark для тех, кто пришёл с Pandas:

```python
df1 = spark.read.csv("big.csv")      # ничего не происходит
df2 = df1.filter("amount > 100")      # ничего не происходит
df3 = df2.groupBy("city").count()     # ничего не происходит

df3.show()                            # ВОТ ТЕПЕРЬ всё считается
```

Spark копит **план**, но НЕ выполняет до **action**.

**Зачем?** Чтобы оптимизатор увидел весь pipeline и переписал его эффективнее:
- Predicate pushdown (фильтр уходит в источник).
- Удаление ненужных колонок.
- Объединение совместимых шагов.
- Замена обычного join на broadcast join.

В Pandas каждая операция выполняется сразу. Поэтому в Pandas вы оптимизируете руками, в Spark — оптимизатор делает это за вас.

## Часть 7. Transformation vs Action

Все операции в Spark делятся на два класса:

| Тип | Что | Примеры |
|-----|-----|---------|
| **Transformation** | Lazy: возвращает новый DataFrame, не считает | `select`, `filter`, `groupBy`, `withColumn`, `join` |
| **Action** | Eager: запускает вычисление | `show`, `count`, `collect`, `write`, `take` |

Запомните это до конца жизни. Если ваш Spark «ничего не делает» — вы забыли action.

## Часть 8. RDD vs DataFrame vs Dataset

Эволюция API:

| Уровень | API | Когда брать |
|---------|-----|-------------|
| Низкий | **RDD** (Resilient Distributed Dataset) | Только если без него никак (редкая обработка непростых форматов) |
| Высокий | **DataFrame** | 90% задач. Это стандарт. |
| Высокий + типы | **Dataset** | Только Scala/Java, в Python = DataFrame |

DataFrame — это **табличный RDD** + схема + Catalyst-оптимизатор. Всегда предпочитайте его.

```python
# RDD (старый низкоуровневый API)
rdd = spark.sparkContext.parallelize([(1, "a"), (2, "b")])
rdd.map(lambda x: (x[0]*10, x[1])).collect()

# DataFrame (современный API)
df = spark.createDataFrame([(1, "a"), (2, "b")], ["id", "name"])
df.withColumn("id2", df.id * 10).show()
```

В курсе мы 95% времени работаем с DataFrame.

## Часть 9. Кластер-менеджеры

Spark может работать в разных режимах. Указывается через `.master(...)`:

| Master | Где | Для чего |
|--------|-----|----------|
| `local[*]` | Один процесс | Учёба, разработка |
| `local[N]` | N потоков | Учебные стресс-тесты |
| `spark://...` | Standalone-кластер | Простые prod-кластера |
| `yarn` | На Hadoop YARN | Классические on-prem |
| `k8s://...` | На Kubernetes | Облако / cloud-native |
| (через Databricks/EMR/Dataproc) | Managed | Production без головной боли |

В курсе используем `local[*]`. На реальных кластерах меняется только `master`.

## Часть 10. Пример минимальной программы

```python
from pyspark.sql import SparkSession

spark = (
    SparkSession.builder
    .appName("MyFirstSpark")
    .master("local[*]")                       # все ядра локально
    .config("spark.driver.memory", "2g")      # 2 ГБ драйверу
    .getOrCreate()
)

df = spark.read.csv("data.csv", header=True, inferSchema=True)
df.printSchema()
df.show(5)
print("Строк:", df.count())

spark.stop()
```

Запускаем как обычный Python-скрипт. Spark всё остальное делает сам.
