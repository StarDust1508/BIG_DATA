# Урок 4.6 — Pandas vs Spark: бенчмарк и выбор

> Лучший Spark-инженер — тот, кто знает, когда Spark **не** нужен. Делаем измеримое сравнение.

---

## Часть 1. Концептуальные различия

| Аспект | Pandas | Spark |
|--------|--------|-------|
| Машина | одна | кластер (или local[*]) |
| Память | RAM, один процесс | RAM + spill на диск, N executor'ов |
| Lazy | нет | да |
| Оптимизатор | нет (вы оптимизируете руками) | Catalyst |
| Стоимость старта | мгновенно | 5–30 секунд (JVM, executor'ы) |
| Размер данных | до 1–10 ГБ | до петабайт |
| Интерактивность | мгновенный отклик | секунды-минуты |

Главный «налог» Spark — **накладные расходы на запуск и shuffle**. На малых данных Spark будет МЕДЛЕННЕЕ Pandas, и сильно.

---

## Часть 2. Где «точка перелома»

Эмпирическое правило (зависит от железа и задачи):

```
Размер  Pandas?    Spark?
< 100 МБ    ✅       ❌ overhead
100 МБ–2 ГБ ✅       ❌ pointless
2–10 ГБ     ⚠️ tune  ✅ начинает иметь смысл
10–50 ГБ    ⚠️ только Polars/Dask на мощной  ✅ хорошо
> 50 ГБ     ❌       ✅ обязателен (или Polars на мега-машине)
```

Это для **одной мощной машины** (16+ ядер, 64+ ГБ RAM). На обычном ноутбуке Spark тоже работает (local[*]), но не даёт особого выигрыша.

## Часть 3. Бенчмарк — сравним сами

См. [практика_3_pandas_vs_spark.py](./практика_3_pandas_vs_spark.py).

Что делает:
1. Генерирует датасет на 10 миллионов строк.
2. Гоняет одинаковую агрегацию на Pandas и PySpark (local[*]).
3. Считает время.

Типичные результаты на ноутбуке 8 ядер / 16 ГБ:

| Задача | Pandas | Spark (local[*]) |
|--------|--------|------------------|
| read 200 МБ CSV | 5–8 с | 10–15 с (старт JVM) |
| read 200 МБ Parquet | 0.5 с | 5–8 с |
| groupBy + sum | 1 с | 3–5 с |
| groupBy + count, P99 | 2 с | 4–6 с |
| 3 × join | 4 с | 8 с |

Видно: на 10М строк **Pandas быстрее**, потому что Spark тратит время на JVM/координацию.

Запустите на 100М (если RAM позволит Pandas) — увидите, как пропорции меняются. На 500М Pandas уже OOM, а Spark спокойно дожимает.

## Часть 4. Принцип «правильного» Spark-кода

Если перешли в Spark не «потому что надо», а **потому что данных реально много** — пишите Spark-стиль, а не «Pandas with overhead»:

### ❌ Pandas-стиль в Spark
```python
# collect → for-loop → удалять колонки → ...
data = df.collect()
out = []
for row in data:
    if row.amount > 100:
        out.append((row.id, row.amount * 2))
```

### ✅ Spark-стиль
```python
df.filter("amount > 100").select("id", (F.col("amount")*2).alias("doubled"))
```

Разница в производительности — **100×** и больше.

## Часть 5. Когда смешивать

Можно. Часто оптимально:

- **Pandas** для интерактивной разведки на сэмпле.
- **Spark** для прода на полных данных.

```python
# Берём сэмпл для исследования
sample = df.sample(0.001).toPandas()
sample.describe()
sns.histplot(sample["amount"])

# Найдя инсайт, пишем код в Spark
df.filter("amount > X").write.parquet(...)
```

Также Pandas UDF и `applyInPandas` — мост в обратную сторону: применить pandas-функцию **внутри** Spark-пайплайна.

## Часть 6. Дерево решений (повторение из 02.6)

```
Размер?
├── < 2 ГБ ─────► Pandas / Polars
├── 2–10 ГБ ────► Polars / DuckDB
├── 10–100 ГБ ──► Polars (мощная) / DuckDB / Spark local[*]
└── > 100 ГБ ──► PySpark на кластере
```

И всегда:
- Если нужен SQL → DuckDB / Spark SQL.
- Если нужен реальный streaming → Spark Structured Streaming / Flink.
- Если нужен ML на распределёнке → Spark MLlib.

## Часть 7. Спорные кейсы

**«У нас 5 ГБ, но в команде уже Spark — что делать?»**
Используйте Spark. Цена «двух стэков» обычно выше пары секунд оверхеда.

**«У нас 50 ГБ один раз, потом 5 МБ ежедневно».**
Один раз Spark, потом Pandas. Не привязывайтесь к инструменту.

**«У нас 200 ГБ Parquet, всё в S3, простые SQL-запросы».**
Возможно, DuckDB будет быстрее и проще Spark.

---

## Часть 8. Стоимость с точки зрения юриста

Юр.отдел и финансовый часто требуют **обоснования технических решений**. Применительно к Big Data:

- Spark-кластер в облаке (даже spot-инстансы) — десятки тысяч ₽/мес.
- Snowflake / BigQuery — поминутная оплата, дешевле для редких запросов.
- DuckDB на ноутбуке — бесплатно для аналитика.

Решение «брать Spark / не брать» часто **финансовое**, а не техническое. Подкрепляйте бенчмарками.
