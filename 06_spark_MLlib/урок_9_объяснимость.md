## Часть 1. Зачем объяснимость

ML-модели часто принимают «значимые» решения: кредит, страхование, найм, медицина.

Регулирование:
- **GDPR Art. 22**: субъект имеет право на human review автоматизированного решения, имеющего значимые последствия.
- **AI Act (ЕС, 2024)**: системы «высокого риска» (HR, кредит, медицина, юстиция) — обязательная документация и объяснимость.
- **152-ФЗ**: общие требования к обработке. Прямой ссылки на ML нет, но если решение влияет на права субъекта — применяются общие правила.

Прикладная сторона: «модель отказала, обоснуй» в суде или в споре с регулятором.

## Часть 2. Уровни объяснимости

### Глобальная

«Как модель работает в целом?»
- Какие признаки самые важные?
- Какие зависимости?
- Какие группы данных модель видит хуже?

**Инструменты в MLlib:**
- Feature importance (RF, GBT).
- Коэффициенты (LR).
- Permutation importance (вручную).

### Локальная

«Почему модель сказала про **этого** клиента "отказ"?»
- Какие признаки внесли вклад?
- Что нужно изменить, чтобы решение стало другим?

**Инструменты:**
- SHAP (не в MLlib, но можно подключить).
- LIME (внешняя библиотека).
- Простые: top-N признаков по `|coef × value|` для линейных моделей.

## Часть 3. Глобальная объяснимость

### Feature importance в RF/GBT
```python
rf_stage = model.stages[-1]
importances = list(zip(feature_names, rf_stage.featureImportances.toArray()))
importances.sort(key=lambda x: -x[1])
for name, imp in importances[:10]:
    print(f"  {name:30s} {imp:.4f}")
```

### Коэффициенты LR
```python
lr_stage = model.stages[-1]
print("Intercept:", lr_stage.intercept)
for name, c in zip(feature_names, lr_stage.coefficients):
    sign = "+" if c > 0 else "-"
    print(f"  {sign} {abs(c):.4f}  {name}")
```

⚠️ Коэффициенты надо смотреть **после стандартизации** — иначе нельзя сравнивать.

### Permutation importance (моделонезависимая)
```python
import random

def permutation_importance(model, df, feature_col, label_col, evaluator):
    base = evaluator.evaluate(model.transform(df))
    pdf = df.toPandas()
    importances = {}
    for col in pdf.columns:
        if col == label_col:
            continue
        shuffled = pdf.copy()
        shuffled[col] = shuffled[col].sample(frac=1).values
        shuffled_df = spark.createDataFrame(shuffled)
        score = evaluator.evaluate(model.transform(shuffled_df))
        importances[col] = base - score
    return importances
```

(Это работает на small/medium датасете через toPandas. На больших — отдельная история.)

## Часть 4. Локальная объяснимость

В MLlib встроенного SHAP нет. Подходы:

### Простой top-N для LR
```python
def explain_one_lr(model, row, feature_names):
    coefs = model.stages[-1].coefficients.toArray()
    contribs = [(name, c, v, c * v)
                for name, c, v in zip(feature_names, coefs, row)]
    contribs.sort(key=lambda x: -abs(x[3]))
    return contribs[:5]

# Применение
row = pred.filter("client_id = 12345").select("features").first()["features"]
top = explain_one_lr(model, row.toArray(), feature_names)
for name, c, v, contrib in top:
    print(f"  {name:25s}  value={v:.2f}  coef={c:.3f}  contrib={contrib:+.3f}")
```

### Через SHAP (вне MLlib)
```python
# pip install shap
import shap
# Конвертация Spark-модели в sklearn-эквивалент → SHAP
# Это сложнее, рассматривайте отдельно от MLlib pipeline
```

Часто подход: **inference в Spark, объяснение в Pandas/sklearn на сэмпле**.

## Часть 5. Model Card — главный документ

Это **карточка модели** — стандарт документации (Google, 2019).

Шаблон:

```markdown
# MODEL CARD — Credit Scoring v1.0

## 1. Назначение
Бинарная классификация: вероятность дефолта по кредиту в 90 дней.

## 2. Архитектура
- Алгоритм: RandomForestClassifier(numTrees=200, maxDepth=10)
- Spark MLlib 3.5.0
- Pipeline: StringIndexer → OHE → VectorAssembler → StandardScaler → RF

## 3. Данные обучения
- Период: 2023-01-01 ... 2026-04-30
- Размер: 1 234 567 строк
- Источник: внутренний DWH, таблица loans_history
- Признаки: возраст, доход, кол-во транзакций, история кредитов, регион
- Метка: фактический дефолт 90+ дней
- Версия данных: hash = a3f2c1d

## 4. Метрики на holdout
- ROC-AUC: 0.81
- PR-AUC: 0.34
- F1 @ threshold=0.3: 0.42
- Precision @ 0.3: 0.51
- Recall @ 0.3: 0.36

## 5. Гарантии справедливости
- Disparate Impact ratio (gender): 0.94 — в пределах нормы (0.8...1.25)
- Disparate Impact ratio (age_band): 0.88

## 6. Ограничения
- Модель НЕ применяется к клиентам < 18 лет.
- Модель НЕ обучалась на данных регионов X, Y → может работать хуже.
- При значительном сдвиге распределения — переобучение.

## 7. Объяснимость
- Глобально: feature importance (см. metrics.json).
- Локально: SHAP-объяснение доступно через API /explain/<application_id>.

## 8. Юр.основа обработки
- 152-ФЗ: согласие клиента (форма Z-12).
- GDPR: legitimate interest (банк), право на возражение Art. 22.
- AI Act: категория «высокий риск» (кредитный скоринг).

## 9. Контакты
- Owner: ml-team@company.com
- Data Steward: dpo@company.com
- Last review: 2026-05-15
```

Этот документ — **минимум**, который должен быть у production-модели в области высокого риска по AI Act.

## Часть 6. Audit log при инференсе

В production:
```python
def predict_with_audit(model, row, request_id):
    pred = model.transform(row)
    log.info(json.dumps({
        "request_id": request_id,
        "model_version": MODEL_VERSION,
        "model_hash": MODEL_HASH,
```

## Часть 7. Bias-тестирование

Простая проверка disparate impact (по гендеру, возрасту, региону):

```python
def disparate_impact(pred, protected_col, positive_col="prediction", positive_value=1):
    rates = (pred.groupBy(protected_col)
        .agg(F.avg((F.col(positive_col) == positive_value).cast("double")).alias("positive_rate"))
        .collect())
    rates = sorted(rates, key=lambda r: r["positive_rate"])
    if len(rates) < 2:
        return None
    return rates[0]["positive_rate"] / rates[-1]["positive_rate"]

di = disparate_impact(pred, "gender")
print(f"DI = {di:.2f}  (норма: 0.8...1.25)")
```

DI ниже 0.8 = модель «помогает» одной группе. Это потенциальная **дискриминация** — серьёзная проблема и юридически, и репутационно.

## Часть 8. Контрольный лист для AI Act high-risk

Если модель попадает в категорию «высокий риск» (HR, кредит, медицина, юстиция, образование):

- [ ] Conformity assessment проведена (внутренняя или сторонняя).
- [ ] Risk management system — есть.
- [ ] Data governance + DPIA — есть.
- [ ] Technical documentation = MODEL CARD + DATA CARD.
- [ ] Record-keeping = audit log на каждый инференс.
- [ ] Transparency: клиент знает, что решение автоматизировано.
- [ ] Human oversight: процедура human review.
- [ ] Robustness: bias-тесты, adversarial-проверки.
- [ ] Регистрация в EU AI database.

В РФ аналогичных требований **пока** нет, но они идут (см. указ № 490 и проекты регулирования ИИ).
