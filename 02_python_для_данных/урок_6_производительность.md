## Часть 1. Где у Pandas «потолок»

Pandas — это **один процесс, одна машина, всё в RAM**. Из этого вытекает:

| Размер данных | Реалистичность с Pandas |
|---|---|
| < 100 МБ | даже не задумывайтесь, всё летает |
| 100 МБ – 2 ГБ | работает, но без неаккуратных `apply` |
| 2 – 10 ГБ | работает на 16+ ГБ RAM, нужны оптимизации |
| 10 – 50 ГБ | работает на «жирной» машине (64+ ГБ), очень аккуратно |
| > 50 ГБ | поднимать PySpark / Polars / Dask |

**Правило большого пальца:** Pandas хочет в 5–10× больше RAM, чем размер файла на диске.

## Часть 3. Оптимизация памяти: `dtype`-тюнинг

Это первое, что нужно сделать. Часто экономит 50–80% памяти.

```python
# Колонка возраста (0..120) занимает int64 = 8 байт.
# Реально хватает int8 = 1 байт.
df["age"] = df["age"].astype("int8")

# Колонки с 5 уникальными категориями — category, а не object.
df["currency"] = df["currency"].astype("category")

# float32 вместо float64, если точности хватает.
df["amount"] = df["amount"].astype("float32")
```

Универсальная функция для авто-уменьшения int:
```python
def shrink_ints(df):
    for col in df.select_dtypes("int").columns:
        c_min, c_max = df[col].min(), df[col].max()
        if c_min >= 0:
            for dtype in ("uint8", "uint16", "uint32"):
                if c_max <= np.iinfo(dtype).max:
                    df[col] = df[col].astype(dtype); break
        else:
            for dtype in ("int8", "int16", "int32"):
                if c_min >= np.iinfo(dtype).min and c_max <= np.iinfo(dtype).max:
                    df[col] = df[col].astype(dtype); break
    return df
```

## Часть 4. Чтение больших файлов

### chunksize — стримом
```python
total_amount = 0
for chunk in pd.read_csv("big.csv", chunksize=500_000):
    total_amount += chunk["amount"].sum()
```

### Только нужные колонки
```python
pd.read_csv("big.csv", usecols=["id", "amount"])
pd.read_parquet("big.parquet", columns=["id", "amount"])
```

### Указать dtype при чтении
```python
pd.read_csv("big.csv", dtype={"id": "int32", "city": "category"})
```

Это часто экономит в разы — Pandas не будет «гадать» тип и завышать его.

## Часть 5. Векторизация vs циклы

Главный закон Pandas: **никаких for-циклов по строкам**.

```python
# ❌ Медленно (минуты на 1М строк)
for i, row in df.iterrows():
    df.loc[i, "z"] = row["x"] + row["y"]

# ✅ Векторно (миллисекунды)
df["z"] = df["x"] + df["y"]

# ❌ apply почти такой же медленный, как iterrows
df["z"] = df.apply(lambda r: r["x"] + r["y"], axis=1)

# ✅ numpy.where для условий
df["bucket"] = np.where(df["amount"] > 1000, "big", "small")

# ✅ np.select для нескольких условий
conds = [df["amount"] < 100, df["amount"] < 1000]
choices = ["tiny", "medium"]
df["bucket"] = np.select(conds, choices, default="big")
```

### Когда `apply` всё-таки нужен

Сложная пользовательская функция, которую нельзя выразить через numpy. Тогда `apply` неизбежен. Но это сигнал: **«вы вышли за зону комфорта Pandas, подумайте о Polars или PySpark»**.

## Часть 6. Альтернативы Pandas

### Polars — Pandas, переписанный с нуля на Rust

```bash
pip install polars
```
```python
import polars as pl
df = pl.read_parquet("big.parquet")
df.filter(pl.col("amount") > 1000).group_by("category").agg(pl.col("amount").sum())
```

- В 5–30× быстрее Pandas на тех же задачах.
- Использует все ядра CPU.
- Lazy mode — план выполнения, как в Spark.
- Близкий по идее API.

**Когда брать:** на одной мощной машине, 5–100 ГБ, без распределёнки.

### Dask — Pandas, расколотый на чанки

```bash
pip install dask
```
```python
import dask.dataframe as dd
df = dd.read_csv("big-*.csv")
df.groupby("city").amount.mean().compute()
```

- API почти 1:1 с Pandas.
- Распараллеливание (несколько ядер или машин).
- Lazy.

**Когда брать:** API Pandas хочется сохранить, но данные не помещаются.

### DuckDB — SQL над Parquet

```bash
pip install duckdb
```
```python
import duckdb
duckdb.sql("SELECT city, SUM(amount) FROM 'big.parquet' GROUP BY city").df()
```

- Невероятно быстрый OLAP-движок.
- Работает прямо по Parquet-файлу без загрузки в RAM.
- SQL.

**Когда брать:** аналитика на 1–100 ГБ Parquet/CSV. Часто **самый быстрый** вариант на одной машине.

### PySpark

Когда:
- Данные > 50 ГБ.
- Нужно распределить на кластер.
- Уже есть Spark-экосистема в проекте.

Цена: накладные расходы на JVM. На малых данных будет **медленнее** Pandas.

## Часть 7. Дерево решений

```
Размер данных?
├── < 2 ГБ ─────► Pandas. Точка.
│
├── 2–10 ГБ ────► Pandas с dtype-тюнингом ИЛИ Polars
│
├── 10–100 ГБ ──► Polars (одна машина) ИЛИ DuckDB ИЛИ Dask
│                   PySpark тоже работает, но избыточен
│
└── > 100 ГБ ──► PySpark, обязательно кластер
```

Бонусные вопросы:

- Нужны транзакции и UPDATE/DELETE? → Lakehouse (Delta/Iceberg).
- Нужно потоково? → Kafka + Spark Streaming / Flink.
- SQL-аналитика без кода? → DuckDB / ClickHouse / BigQuery.

## Часть 8. Когда PySpark **проигрывает** Pandas

Не каждая «большая» задача — про Spark.

| Сценарий | Pandas / Polars | PySpark |
|----------|------------------|---------|
| 5 ГБ, на одной 32 ГБ машине | ✅ быстрее | ❌ overhead |
| Множество мелких файлов | ✅ | ❌ медленный startup |
| Интерактивная разведка | ✅ | ❌ долгий feedback |
| Прод-пайплайн 500 ГБ ежедневно | ❌ не вытянет | ✅ |
| Распределённая ML на кластере | ❌ | ✅ |

Это очень важный психологический момент: **«Big Data» — не цель, а вынужденная мера**. Если можно Pandas — берите Pandas.
