# Урок 1.3 — Распределённые вычисления: CAP, ACID/BASE, MapReduce

> Самый «теоретический» урок курса. Но без него Spark — это «магия с непредсказуемыми падениями».

---

## Часть 1. Зачем вообще распределённые системы

Один сервер имеет границы:
- **CPU**: даже топовые 128 ядер.
- **RAM**: ≤ 6 ТБ на премиум-машинах, в реальности ~256 ГБ.
- **Диск**: тёрабайты, но скорость одного NVMe — около 7 ГБ/с.
- **Сеть**: 100 Гбит/с в датацентре.
- **Надёжность**: один сервер падает.

**Решение:** взять много дешёвых машин и заставить их работать как одну. Это даёт:
- горизонтальное масштабирование («можем добавить ещё машин»);
- отказоустойчивость (одна упала — система живёт);
- ниже стоимость на 1 ГБ обработанных данных.

Цена:
- **сложность**. Распределённая система — это всегда сложнее, чем «один сервер посильнее».

## Часть 2. Теорема CAP

Эрик Брюер, 2000 год. Одна из самых цитируемых идей в Big Data.

> В распределённой системе **невозможно одновременно** гарантировать все три свойства:
> - **C**onsistency (согласованность) — все узлы видят одни и те же данные в один и тот же момент.
> - **A**vailability (доступность) — каждый запрос получает ответ (пусть не самый свежий).
> - **P**artition tolerance (устойчивость к разделению сети) — система работает, даже если связь между узлами пропала.

В реальности сеть **обязательно** иногда «разрывается» (пакеты теряются, узлы недоступны). Значит P нужно поддерживать всегда. Выбор между **CP** и **AP**:

| Тип | Что предпочитает | Примеры |
|-----|-----------------|---------|
| **CP** | Согласованность над доступностью | HBase, MongoDB (по умолч.), Zookeeper |
| **AP** | Доступность над согласованностью | Cassandra, DynamoDB, CouchDB |

**Пример «на пальцах»:** есть две ноды. Сеть между ними оборвалась.
- В CP-системе одна из нод откажется отвечать (чтобы не выдать устаревшие данные) → меньше доступность, но согласованно.
- В AP-системе обе будут отвечать (возможно разное) → доступно, но временно несогласованно.

**Юридическая параллель:** банк выбирает **CP** (нельзя «продать одну и ту же акцию двум»); соцсеть — **AP** (если ваш лайк появится на 2 секунды позже, никто не пострадает).

## Часть 3. ACID vs BASE

Старая школа БД (PostgreSQL, Oracle, MySQL):

**ACID** — Atomicity, Consistency, Isolation, Durability.
Транзакции либо целиком, либо никак. Жёсткие гарантии.

Новая школа NoSQL и Big Data:

**BASE** — Basically Available, Soft state, Eventual consistency.
«Когда-нибудь все согласуются». Зато можно горизонтально масштабироваться.

```
                  Шкала гарантий
   Слабые                                Сильные
       │                                     │
       ▼                                     ▼
  Eventual ── Causal ── Read-your-writes ── Linearizable
```

**Тренд 2020-х:** многие распределённые БД (Spanner от Google, CockroachDB) **возвращают** ACID-подобные гарантии за счёт сложного консенсуса. Граница размывается.

## Часть 4. Идея MapReduce

Эта идея — мозг почти всего Big Data, даже современный Spark наследует её.

### Пример: посчитать слова в книгах

У вас 1 миллион книг, распределённых по 100 серверам. Задача — сосчитать, сколько раз встречается каждое слово.

**Подход 1 (наивный):** «слить всё в один файл и посчитать». 1 ПБ копировать по сети → нереально.

**Подход 2 (MapReduce):**

```
Шаг 1: MAP (на каждом сервере параллельно)
   Книга1 → [(слово1, 1), (слово2, 1), (слово1, 1), ...]
   Книга2 → [(слово3, 1), (слово1, 1), ...]
   ...
   100 серверов работают параллельно.

Шаг 2: SHUFFLE
   Все пары (слово, число) перегруппировываются по ключу:
   "слово1" → [1, 1, 1, 1, ...] (на одном узле)
   "слово2" → [1, 1] (на другом)

Шаг 3: REDUCE (параллельно по ключам)
   "слово1" → sum([1, 1, 1, 1]) = 4
   "слово2" → sum([1, 1]) = 2
```

Победа: каждая машина пилит свою долю; сеть напрягается только во время shuffle.

### MapReduce на Python (учебно, без Hadoop)

`mapper.py`:
```python
import sys
for line in sys.stdin:
    for word in line.lower().split():
        print(f"{word}\t1")
```

`reducer.py`:
```python
import sys
from collections import defaultdict

counts = defaultdict(int)
for line in sys.stdin:
    word, n = line.strip().split("\t")
    counts[word] += int(n)
for w, c in sorted(counts.items()):
    print(f"{w}\t{c}")
```

Запуск (без Hadoop, в обычном shell):
```bash
cat books.txt | python mapper.py | sort | python reducer.py
```

В Hadoop — то же самое, но `sort` происходит распределённо, и `mapper`/`reducer` бегут на сотнях машин одновременно.

## Часть 5. Почему Spark в 10–100 раз быстрее MR

MR пишет промежуточные результаты на диск **между каждым шагом**. Если у вас цепочка из 10 операций — 10 раз туда-сюда HDD.

Spark делает то же самое в **памяти** (RAM). Промежуточные RDD кешируются. Это даёт основной выигрыш.

Дополнительно у Spark:
- **DAG-оптимизация** (объединяет совместимые шаги).
- **Catalyst** для DataFrame (оптимизатор уровня SQL).
- **Tungsten** (бинарная in-memory сериализация).

Поэтому современная индустрия пишет на Spark, а MapReduce остался только в наследии.

## Часть 6. Локальность данных (Data Locality)

Сетевой трафик дороже CPU. Поэтому Hadoop/Spark стараются **посылать код к данным**, а не наоборот.

```
HDFS блок X лежит на DataNode 7.
Spark назначает task для блока X → запускает executor на DataNode 7.
Нет сетевой передачи блока — компьют едет к данным.
```

Это работает только если файловая система и compute «знают друг про друга». В облаке (S3) этого знания нет — данные тянутся по сети. Поэтому облачный Spark обычно медленнее on-prem на тех же объёмах, но **дешевле** и проще в эксплуатации.

## Часть 7. Failure — это норма

В кластере из 1000 машин **каждый день что-то падает**. Жёсткие диски сыпятся, сетки моргают, ноды перезагружаются. Если ваш ETL падает из-за одной упавшей машины — он не для прода.

Hadoop/Spark обеспечивают отказоустойчивость через:

1. **Репликацию данных** (HDFS по умолчанию хранит 3 копии).
2. **Перезапуск тасков** на других нодах.
3. **Lineage в Spark** — RDD помнит, как был построен, и при потере перевычисляется.

Для юриста: это значит, что **«удалить запись»** в распределённой системе технически означает **удалить во всех копиях, ВО ВСЕХ резервных копиях, и потенциально перезапустить все downstream-пайплайны**. Поэтому право на забвение — это не «нажать DELETE».
