# Урок 1.4 — Форматы хранения: CSV, JSON, Parquet, ORC, Avro

> Один и тот же датасет можно хранить в разных форматах. Выбор формата влияет на размер в 10×, скорость чтения в 100×, и удобство эволюции схемы.

---

## Часть 1. Две большие категории

```
ФОРМАТЫ
├── СТРОКОВЫЕ (row-oriented)
│   └── Хранят записи целиком, одна за другой.
│       CSV, JSON, Avro
│
└── КОЛОНОЧНЫЕ (columnar)
    └── Хранят значения одной колонки рядом.
        Parquet, ORC
```

### Визуально

Таблица:
```
id │ name  │ age │ city
 1 │ Аня   │ 25  │ Москва
 2 │ Боря  │ 31  │ Питер
 3 │ Витя  │ 19  │ Казань
```

**Строковое хранение (на диске):**
```
1, Аня, 25, Москва | 2, Боря, 31, Питер | 3, Витя, 19, Казань
```

**Колоночное хранение (на диске):**
```
[ids:    1, 2, 3]
[names:  Аня, Боря, Витя]
[ages:   25, 31, 19]
[cities: Москва, Питер, Казань]
```

Почему это важно:

| Сценарий | Что лучше |
|---|---|
| «Прочитать одну запись целиком» (OLTP) | Строковое |
| «Посчитать средний возраст из 100 млн строк» | Колоночное (читаем только колонку `age`) |
| «Сжать данные» | Колоночное (соседние значения похожи) |
| «Дописывать по одной строке» | Строковое |

В Big Data **подавляющее большинство** — это аналитика → колоночное хранение → **Parquet**.

## Часть 2. Сравнительная таблица

| Формат | Тип | Схема | Сжатие | Эволюция схемы | Где используется |
|--------|-----|-------|--------|----------------|------------------|
| **CSV** | row | нет (на чтение) | внешнее (gzip) | плохо | обмен, экспорт |
| **JSON** | row | гибкая | внешнее | хорошо | API, логи |
| **Avro** | row | strict schema | встроенное | отлично | Kafka, ingestion |
| **Parquet** | columnar | strict schema | встроенное (snappy/gzip) | хорошо | Spark, аналитика |
| **ORC** | columnar | strict schema | встроенное | хорошо | Hive |

## Часть 3. Когда какой формат

### CSV
- ✅ Универсальный, читают все.
- ❌ Нет типов (всё строка), нет схемы.
- ❌ Не сжимается компактно.
- ❌ Никаких null/массивов.
- **Когда использовать:** только для обмена или маленьких файлов.

### JSON
- ✅ Гибкий, поддерживает вложенность.
- ❌ Огромная избыточность (повторение ключей в каждой строке).
- **Когда:** логи, API. Внутри пайплайна — конвертировать в Parquet.

### Avro
- ✅ Строковый бинарный, schema-based.
- ✅ Schema evolution: можно добавить поле и читать старые файлы.
- **Когда:** Kafka-сообщения, ingestion-формат.

### Parquet
- ✅ Колоночный → быстрая аналитика.
- ✅ Встроенное сжатие (snappy по умолчанию).
- ✅ Predicate pushdown: «прочитать только нужные строки».
- ✅ Колонка не читается, если не выбрана.
- **Когда:** 95% случаев в Spark.

### ORC
- ✅ Аналог Parquet, родился в Hive.
- Чуть лучшие компрессия и индексы для некоторых задач.
- **Когда:** если стек завязан на Hive/Hortonworks.

## Часть 4. Сжатие: что выбрать

Все колоночные форматы поддерживают разные кодеки:

| Кодек | Сжатие | Скорость |
|-------|--------|----------|
| **snappy** | средне | быстро ✅ (по умолч.) |
| **gzip** | хорошо | медленнее |
| **zstd** | очень хорошо | быстро (новый) |
| **lz4** | слабее | очень быстро |
| **brotli** | максимум | медленно |

Для Spark по умолчанию `snappy` — отличный компромисс.

## Часть 5. Эволюция схемы

Реальные данные **меняются**. Сегодня поле `email`, завтра добавили `phone`, послезавтра убрали `fax`.

| Формат | Что умеет |
|--------|-----------|
| CSV | Ничего. Поломалось — переписывайте. |
| JSON | Гибко, но без гарантий. |
| Avro | Schema registry, чёткие правила (default values). |
| Parquet | Может добавлять колонки, читать старые. |

**Best practice:** в Kafka → Avro со Schema Registry. В Data Lake → Parquet с метаданными.

## Часть 6. Партиционирование (Hive-style)

Не путать с партициями Spark в памяти. Это **физическое** разбиение на диске по значению колонки:

```
/data/transactions/
   year=2024/
      month=11/
         part-00000.parquet
         part-00001.parquet
      month=12/
         part-00000.parquet
   year=2025/
      ...
```

Когда вы делаете `WHERE year = 2025`, Spark **не читает** папки `year=2024`. Это **partition pruning**.

В PySpark:
```python
df.write.partitionBy("year", "month").parquet("/data/transactions")
```

⚠️ **Правило большого пальца:** партиции должны быть «средними» — не 10 файлов по 1 ПБ и не 10 млн файлов по 1 КБ. Идеал — десятки/сотни МБ на партицию.

## Часть 7. Бакетирование (опционально)

Похоже на партиционирование, но фиксированное число «корзин» по hash(ключа). Используется для:
- ускорения join'ов (одинаковые ключи всегда в одной корзине);
- стабильности по числу файлов.

```python
df.write.bucketBy(100, "user_id").sortBy("ts").saveAsTable("events")
```

## Часть 8. Lakehouse-форматы

Сверху Parquet нарастили слой метаданных и транзакционности:

| Формат | От кого | Особенности |
|--------|---------|-------------|
| **Delta Lake** | Databricks | ACID, time travel, MERGE |
| **Iceberg** | Netflix / Apache | hidden partitioning, snapshot isolation |
| **Hudi** | Uber / Apache | upsert, incremental ingestion |

Они дают:
- **ACID транзакции** в Data Lake.
- **DELETE / UPDATE** (для GDPR-compliance).
- **Time travel:** «как выглядела таблица 3 дня назад».
- **Schema evolution** на уровне таблицы.

В курсе мы потрогаем Delta в модуле 05.

## Часть 9. Юридический угол

«Право на забвение» (Art. 17 GDPR) и Parquet:

- В голом Parquet **нельзя** удалить одну строку — файлы иммутабельны.
- Решение: перезаписать партицию **без** удалённого пользователя.
- Затратно, но возможно.
- Lakehouse-форматы (Delta/Iceberg/Hudi) делают это **встроенным** `DELETE FROM table WHERE user_id = X`.

Поэтому если в проекте есть ПДн и работа с GDPR/152-ФЗ — Lakehouse-формат предпочтительнее голого Parquet.
