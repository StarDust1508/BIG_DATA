## Часть 1. Краткая история

```
2003 — Google File System paper
2004 — MapReduce paper (Google)
2006 — Apache Hadoop (HDFS + MR), отделение от Nutch
2008 — Cassandra (Facebook)
2009 — MongoDB (10gen)
2010 — Apache Spark (UC Berkeley AMPLab)
2011 — Kafka (LinkedIn)
2012 — YARN (Hadoop 2.0): MR — не единственный compute
2013 — Apache Flink (TU Berlin)
2015 — Spark DataFrame; уход от чистого RDD
2016 — Snowflake публичный запуск (DWH в облаке)
2017 — Kubernetes становится стандартом → cloud-native data
2019 — Delta Lake (Databricks); рождение Lakehouse
2020 — Apache Iceberg, Apache Hudi
2024 — EU AI Act
```

Тренд: **«дешёвое железо + open source + облако»**. Раньше нужна была своя серверная — теперь поднимаете кластер за 10 минут в Databricks/EMR.

## Часть 3. По компонентам — что и зачем

### 🗄 Хранилище

| Технология | Что | Когда брать |
|------------|-----|-------------|
| **HDFS** | Распределённая ФС из Hadoop | On-prem, классический Hadoop |
| **S3 / GCS / Blob** | Облачное объектное хранилище | Облако, де-факто стандарт |
| **MinIO** | S3-совместимое self-hosted | Локально, в k8s |
| **Cassandra** | NoSQL, оптимизирована для записи | Высокий throughput на запись, low-latency чтения по ключу |
| **HBase** | NoSQL поверх HDFS | Random access к большим таблицам |
| **MongoDB** | NoSQL документная | Полуструктурированные данные |
| **Snowflake / BigQuery / Redshift** | Облачные DWH | SQL-аналитика, без управления кластером |

### ⚙ Compute

| Технология | Что | Когда |
|------------|-----|-------|
| **Hadoop MapReduce** | Классика batch | Только наследие. Новое не пишут. |
| **Spark** | Универсальный compute (batch + streaming) | Большинство задач |
| **Flink** | True streaming | Когда latency критична |
| **Presto / Trino** | Распределённый SQL-движок | Интерактивные запросы к Data Lake |
| **Hive** | SQL поверх MR/Tez/Spark | Стандарт для DWH на Hadoop |

### 📬 Ingestion

| Технология | Что |
|------------|-----|
| **Kafka** | Распределённый брокер сообщений (де-факто стандарт) |
| **Kinesis** | AWS-аналог Kafka |
| **Pub/Sub** | GCP-аналог |
| **NiFi** | Визуальный data flow |
| **Debezium** | Change Data Capture (CDC) из БД |

### 🎯 Orchestration

| Технология | Что |
|------------|-----|
| **Airflow** | DAG-оркестратор, де-факто стандарт |
| **Dagster** | Современный, лучше типизация |
| **Prefect** | Лёгкий аналог Airflow |
| **dbt** | Трансформации в DWH |

## Часть 4. Lakehouse — что это и почему важно

Раньше выбирали между:

- **Data Warehouse** (DWH) — структурированно, SQL, дорого, мало форматов. Snowflake, BigQuery.
- **Data Lake** — всё подряд, S3+Parquet, дёшево, но никаких ACID, никаких транзакций.

**Lakehouse** = DWH + Data Lake в одном:
- хранение в S3+Parquet (дёшево),
- + транзакции, UPDATE, DELETE, time-travel (как в DWH).

Реализуют:
- **Delta Lake** (Databricks)
- **Apache Iceberg** (Netflix → Apache)
- **Apache Hudi** (Uber → Apache)

Для юриста: Lakehouse — это **единственный** реалистичный способ соблюдать «право на забвение» в Data Lake. См. [правовые_аспекты.md §5.1](../ресурсы/правовые_аспекты.md).

## Часть 5. Типичные «стэки» компаний

Чтобы видеть, как из кубиков собирают реальные системы.

### Малый бизнес (≤ 100 ГБ данных)
```
PostgreSQL → dbt → Metabase
```
Никакой Big Data. Однажды это перестанет помещаться — тогда мигрируют.

### Средний (1–100 ТБ)
```
Источники → Kafka → Spark → S3+Parquet → Snowflake → BI
                                  ↑
                              Airflow
```

### Крупный (петабайты)
```
Kafka → Flink (real-time) ──┐
                            ├→ Data Lake (S3 + Iceberg) → Trino → BI
Batch ETL → Spark ──────────┘                                ↓
                                                          MLlib
                                                             ↓
                                                  модели в production
```

### Облачно-нативный
```
Источники → Cloud Functions → BigQuery → Looker
```
Никакого Spark — всё в managed-сервисах.

## Часть 6. На что смотреть, когда «знакомитесь» с чужим стэком

Шесть вопросов, которые быстро дают картину:

1. **Где хранится?** S3? HDFS? Snowflake?
2. **Чем считается?** Spark? dbt? Чистый SQL?
3. **Как туда попадает?** Batch? Kafka?
4. **Как оркестрируется?** Airflow? cron?
5. **Что наверху?** Tableau? Notebook?
6. **Какие требования к latency и качеству?**

Дальше всё — детали.

## Часть 7. Что НЕ Big Data, но часто рядом

Эти технологии **не** про большие данные сами по себе, но их вы будете встречать в Big Data проектах:

- **PostgreSQL, MySQL** — источники данных и метаданные.
- **Redis** — кэш и стейт.
- **Elasticsearch** — полнотекстовый поиск по логам.
- **Grafana + Prometheus** — мониторинг.
- **Jupyter / VS Code** — рабочее место аналитика.
